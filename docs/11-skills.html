Skills

Disk-loaded, model-invocable prompt packs. A skill is a single SKILL.md with YAML frontmatter and a markdown body, sitting under <configDir>/skills/<name>/. The model discovers skills through a cache-friendly skill_listing attachment and invokes them by name via the new Skill tool. The substituted body comes back as the tool result, and the model acts on it on the next turn.

Concept

Skills give you a way to ship reusable model behavior — “commit”, “deploy”, “security review” — as plain markdown files that the model can find and invoke on demand. There is no plugin protocol, no JS code path to register, no fork or subagent: the body is substituted in-place and returned to the model as a tool result. The model reads the instructions and follows them using the existing tools.

Skills are intentionally narrow in v1: inline execution only. There is no fork mode, no MCP-sourced skills, no shell injection in the body, no path-conditional skills, no programmatic registration, no user-home fallback. Skawld looks for skills in one place — <configDir>/skills/ — and that is the whole contract.

File format

A skill is a directory: <configDir>/skills/<name>/SKILL.md. The file starts with a YAML frontmatter block, followed by a markdown body.

---
name: commit
description: Stage changes and create a commit.
when_to_use: When the user asks to commit changes.
arguments:
  - message
argument_hint: "<message>"
allowed_tools:
  - Bash
model: claude-sonnet-4-6
version: "1.0.0"
disable_model_invocation: false
---

Run `git status` to see what changed, then `git commit -m "$message"`.
You may also reference files in ${SKAWLD_SKILL_DIR}/templates/ if present.

Frontmatter fields

Key (YAML)TS fieldRequiredMeaning
namenamenoDefaults to the dir name. Must match /^[a-z0-9][a-z0-9_-]*$/i. If both name and dir name are present and disagree, the frontmatter wins.
descriptiondescriptionyesOne-line summary surfaced in skill_listing.
when_to_usewhenToUsenoOptional hint appended to the listing entry as — <when_to_use>.
allowed_toolsallowedToolsnoTool names that will be additively allowed for the one assistant turn that follows the Skill invocation. Existing deny rules still win.
argumentsargumentsnoNamed argument slots, substituted positionally from the shell-split args string. Each name must match /^[a-z_][a-z0-9_]*$/i; "ARGUMENTS" is reserved.
argument_hintargumentHintnoFree-form hint appended to the listing entry as (args: <hint>).
modelmodelnoPer-skill model override for the one assistant turn that follows. The [1m] extended-context suffix on the session model is carried forward automatically when missing.

disable_model_invocationdisableModelInvocationnoDefault false. When true the skill is hidden from skill_listing and rejected if the model tries to invoke it anyway.

Unknown frontmatter keys are rejected at load time: the offending skill is logged and skipped (it never throws). Same for missing description, malformed YAML, and dir/skill-name collisions.

Subfolder assets & progressive disclosure

A skill's directory is just a directory — you can drop any extra files alongside SKILL.md (templates, scripts, reference docs). They are not loaded automatically. The model can read them via the existing Read / Glob / Bash tools by referencing ${SKAWLD_SKILL_DIR}, which is substituted into the body with the absolute skill-directory path.

This is the “progressive disclosure” pattern: keep the skill_listing entries short, keep the SKILL.md body terse, and put heavyweight context in sibling files that the model only loads when it actually invokes the skill.

Substitution

When the model invokes the Skill tool, skawld substitutes the body in one pass before returning it as the tool result. Substitution order (named args first, then reserved variables):

  1. Named args. The args string passed by the model is shell-split (single + double quotes, backslash escape). Each token is mapped positionally to the arguments slots. Each $<name> in the body becomes that token, or the empty string if the slot is unmapped.
  2. $ARGUMENTS. Replaced with the raw (un-split) args string, quotes preserved.
  3. ${SKAWLD_SKILL_DIR}. Replaced with the absolute path to the skill's directory.
  4. ${SKAWLD_SESSION_ID}. Replaced with the current session id.

Substituted bodies are prepended with a single header line:

Skill base directory: /abs/path/to/skill

<body with substitutions applied>

Substitution is single-pass — post-substitution text is not re-scanned. There is no ${ENV_VAR} support for arbitrary environment variables; only the two skawld-defined ones above. There is no shell injection (no backtick-command syntax).

Discovery: skill_listing

When at least one skill is loaded, skawld prepends a single text content block to the first user message of every brand-new session. The block is wrapped in <system-reminder> and contains the skill_listing:

<system-reminder>
<skill_listing>
- commit: Stage changes and create a commit. — When the user asks to commit changes. (args: <message>)
- deploy: Trigger a deploy via the configured shell script.
</skill_listing>
</system-reminder>

The listing is built once at Agent construction and reused unchanged for every turn — same bytes, same prompt cache entry. The text is sorted by name (case-insensitive) and budgeted to roughly 1% of the model's context window (fallback: 8000 chars). When the total fits, full entries are emitted; otherwise descriptions are equal-allocated and truncated with a trailing …. If the budget drops below 20 chars per description, the listing falls back to names-only.

Skills with disable_model_invocation: true are omitted from the listing.

Invocation: the Skill tool

The model invokes a skill by calling the Skill tool:

{
  "tool": "Skill",
  "input": { "skill": "commit", "args": "fix bug" }
}

Behavior:

  • A leading / on the skill name is stripped (so "/commit" resolves to commit).
  • Unknown skill name → tool result with is_error: true, content Unknown skill: <name>.
  • Skill marked disable_model_invocation: true → tool result with is_error: true.
  • Otherwise: the substituted body is returned as the tool result content, and the invocation is recorded on SessionInternal.invokedSkills and persisted via SessionStore.setInvokedSkills.

Skill invocations are sequential — the tool sets parallelSafe: false, so the scheduler serializes any concurrent Skill tool_use blocks the model emits.

Permissions

The Skill tool has scope: "exec", so by default in default mode it requires permission. To keep the “informational” skill ergonomic, skawld pre-registers auto-allow rules for skills without allowed_tools and without model overrides:

{ kind: "tool", tool: "Skill", arg: "<skillName>", decision: "allow" }

The arg field on ToolRule is new in v1.skills. It is only meaningful for the Skill tool, where it matches input.skill (use "*" to match any skill name). For all other tools, arg is ignored. User-provided rules are evaluated in order before mode defaults, so an explicit user rule still wins over the auto-allow.

Skills with allowed_tools or a model override do not get an auto-allow rule. They go through the engine's normal “ask” path (or whatever the user's rules dictate).

Per-turn overrides

When a skill declares allowed_tools or model, the SkillTool stashes a SkillOverlay on SessionInternal.pendingSkillOverlay. The loop consumes the overlay immediately before the next provider.send and clears it in a try/finally:

  • Scope. One assistant turn only — applied to the very next send, plus the tool execution that follows. Cleared before any subsequent turn's send.
  • allowedTools. The scheduler short-circuits an ask decision to allow when the call's tool name is in the overlay set. Existing deny rules still win above. The overlay is additive; it does not mutate the engine's stored rules.
  • modelOverride. Replaces session.model for the very next provider.send. If a ContextLengthError forces a compaction retry mid-turn, the retry uses the session's default model (the override is spent on the first attempt).
  • [1m] suffix. resolveModelOverride(override, sessionModel) carries the [1m] extended-context suffix from the session model when the override lacks one.

Compaction survival

Skill bodies are substantial — letting them get summarized away on compaction would defeat the purpose. After every compaction, the loop:

  1. Pushes a fresh user message with the skill_listing block (byte-identical to the original).
  2. Pushes one user message per invokedSkills entry, content = the substituted body wrapped in <system-reminder>, in invocation order.

These re-emitted messages are pushed directly to providerView (in-memory) and are not appended to the store. On resume, skawld replays the full pre-compaction message history from the store and reloads invokedSkills from the new sqlite column — so any post-resume compaction will have an authoritative list to re-emit.

SQLite schema

The sessions table gains an invoked_skills_json TEXT NOT NULL DEFAULT '[]' column. PRAGMA user_version is set to 1 on every open and the constant SCHEMA_VERSION = 1 is exported from src/sessions/sqlite-helpers.ts.

There is no migrator. The SDK is unpublished — no existing databases need migrating. If you hit a schema error on a stale local dev DB, delete it:

rm ~/.skawld/sessions.db  # or .skawld/sessions.db in your project root

Public API additions

Importable from "skawld":

  • AgentOptions.configDir?: string — defaults to ".skawld" resolved against cwd. Pass an absolute path to override.
  • SkillError — SkawldError subclass with kind: "skill" and an optional skillName. Thrown only on programmer errors; per-skill load failures become SkippedSkill entries inside the loader, not exceptions.
  • SkillsLoadedEvent — { type: "skills_loaded", skills: Array<{ name, description, when_to_use?, argument_hint? }> }. Emitted once per session, immediately after SystemEvent, only when at least one skill loaded.
  • SkillInvokedEvent — { type: "skill_invoked", name, args?, model_override?, allowed_tools? }. Emitted just before tool_call_start for the Skill tool.
  • SkillCompletedEvent — { type: "skill_completed", name, is_error }. Emitted just after tool_call_end for the Skill tool.

The three skill events are part of the public Event union. The CLI module (08) does not yet exist as a runtime, so these events are observable today only via the programmatic SDK (the for await loop over session.run()). The CLI relay will land alongside CLI module 08.

Deferred to v2

  • paths: conditional skills. Skills are loaded unconditionally in v1; no per-cwd filtering.
  • Fork mode. No way to execute a skill in a subagent or fresh subprocess. Skills are always inline.
  • Shell injection. No !`cmd` syntax in skill bodies.
  • MCP-sourced skills. Skills come from disk only; no MCP server can expose skills.
  • User-home fallback. Only <configDir>/skills/. There is no ~/.skawld/skills/ overlay.
  • Programmatic registration. No agent.registerSkill(...). The disk loader is the only source.
  • CLI event relay. Deferred to CLI module 08 (which itself is still pending).

  • Agent + Session wiring: configDir option, lazy load, conditional SkillTool registration, listing cache, skill events.

  • SQLite invoked_skills_json column + PRAGMA user_version = 1; SessionStore.setInvokedSkills on both SqliteSessionStore and InMemorySessionStore.
