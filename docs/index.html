skawld

An open-source TypeScript agent harness for software engineering. Pluggable LLM providers, a small set of high-leverage built-in tools, a granular permission model, persistent sessions, and a clean SDK that an interactive CLI will later layer on top of.

What this project is

skawld is an agent loop — an LLM driving a small set of tools (file editing, shell, search) to accomplish software-engineering tasks autonomously. Think of it as a Claude Code-style harness, written in TypeScript, designed from day one to be embedded in other applications as much as run as a CLI.

The non-negotiable design goals:

  • Provider-agnostic. Anthropic and OpenAI (both Chat Completions and Responses) work in v1. The provider base classes are designed so OpenAI-compatible endpoints (Ollama, vLLM, Groq, DeepSeek, etc.) and Anthropic-compatible endpoints (Bedrock, Vertex) can be added later by subclassing.
  • Embeddable. The TypeScript SDK is a direct import — import { Agent } from 'skawld' — with no IPC layer between the developer and the engine. Custom session stores, compaction strategies, and permission handlers are plain TypeScript objects passed into the Agent constructor.
  • Predictable. Typed events, typed errors, deterministic permission evaluation, no hidden global state.
  • Open source-friendly. Every subsystem is a class or interface that contributors can extend without inventing a plugin protocol.

Architecture at a glance

The codebase is a single npm package (skawld) with a layered internal structure. The core engine has zero I/O assumptions — it does not write to stdout, does not handle terminals, and emits everything as typed events. The CLI is just one consumer of this engine; the SDK is another (and they are the only two in v1).

Module map

Each module below maps to a folder in src/ (created during implementation) and has a dedicated doc page.

Doc src/ folder Purpose
00 Foundations src/core/ (types, events, errors) Shared type definitions, event union, typed error hierarchy. Nothing depends on anything else; everything depends on this.
01 Providers src/providers/ BaseProvider plus concrete implementations for Anthropic, OpenAI Chat Completions, and OpenAI Responses.
02 Tools src/tools/ Tool interface plus the ten built-in tools: Read, Write, Edit, Bash, Glob, Grep, TaskCreate, TaskList, TaskGet, TaskUpdate.
03 Permissions src/permissions/ Three permission modes (Default, AcceptEdits, YOLO), rule types (tool allow/deny, path scoping, bash patterns), and the canUseTool callback contract.
04 Sessions src/sessions/ SessionStore interface and the default SQLite implementation. Message persistence, task persistence, and resume.
05 Agent & loop src/core/agent.ts, src/core/session.ts, src/core/loop.ts, src/core/compaction.ts The Agent and Session classes, the model→tool orchestration loop, parallel tool execution, streaming, abort handling, and compaction at 80% of the context window.
06 System prompt src/core/system-prompt.ts Cache-optimized prompt assembly. Stable prefix in the system message, dynamic content (date, etc.) in the first user message.
07 SDK src/sdk.ts Public API surface: what gets exported, the async-iterator event shape, error types, and the exports map in package.json.
08 CLI src/cli/ Non-interactive CLI binary. NDJSON events on stdout, JSON control messages on stdin, exit codes. Interactive mode is a future milestone.
09 Config src/config/ Project and user config files (.skawld/config.json), schema, and precedence rules.
10 Project setup repo root package.json (including the exports map), tsconfig.json, bun build + tsc --emitDeclarationOnly for the build, bun test for the test runner, the public subpath surface, and testing conventions.
11 Skills src/skills/, src/tools/skill.ts Disk-loaded SKILL.md prompt packs the model discovers via skill_listing and invokes via the Skill tool. Inline-only, compaction-safe, with one-turn allowed_tools + model overlays.

Reading and build order

Read the docs in numeric order. Implement in roughly the same order, with one inversion: scaffold the project first so you have somewhere to put code.

  1. 10 — Project setup. Scaffold package.json, tsconfig.json, the bun build + tsc declaration pipeline, and the src/ layout. Get an empty entry point building and an empty test suite running under bun test.
  2. 00 — Foundations. Define shared types, the event union, and the error hierarchy. These names are referenced by every other module.
  3. 01 — Providers. Implement BaseProvider, then the three concrete providers. Each provider has its own unit tests that fake the wire layer.
  4. 04 — Sessions. Implement the SessionStore interface, task persistence types, and the SQLite default. Includes the schema migration on first open.
  5. 02 — Tools. Implement the Tool interface and each built-in tool. Task tools depend on the session store from the previous step.
  6. 03 — Permissions. Implement the permission engine. Wire it into the tool execution path (the agent loop will use it in step 8).
  7. 06 — System prompt. Implement the prompt assembler with cache-control breakpoints.
  8. 05 — Agent & loop. This is where everything comes together: the Agent class, the Session class, the model→tool loop, parallel tool execution, streaming, abort handling, compaction.
  9. 07 — SDK. Add the public re-exports. Confirm the exports map matches.
  10. 09 — Config. Implement config loading and the precedence rules.
  11. 08 — CLI. Implement the non-interactive CLI on top of the SDK.
  12. 11 — Skills. Add the disk-loaded skill subsystem on top of the engine. Reuses the existing tool, permission, and compaction machinery.

Glossary

Names used consistently throughout the docs.

TermMeaning
AgentThe top-level skawld object. Constructed once with provider, tools, permissions, system prompt, session store, etc. Used to create or resume sessions.
SessionA persistent conversation, identified by a session id. Contains a message history and exposes run(prompt) which returns an async iterator of events.
RunA single user prompt → agent works → final result cycle inside a session. One iteration of session.run().
TurnA single model invocation: send all messages → receive one assistant message (with text, tool calls, and/or thinking). A run is a sequence of turns until the model stops calling tools.
ProviderA class that knows how to talk to a specific LLM API. Translates skawld's normalized message shape to and from the provider's wire format.
ToolA capability the model can invoke, with a JSON schema, an execute function, and a permission scope (read / write / exec).
Permission modeOne of "default", "acceptEdits", "yolo". Coarse-grained gate before any per-tool rule evaluation.
canUseToolA callback the developer passes to the Agent. Called once per pending tool call when the permission engine cannot resolve a decision from rules alone.
SessionStoreThe persistence interface for messages and task state. Default implementation uses SQLite. Implementations exist behind the same interface for swap-out (Postgres, Mongo, in-memory).
EventA typed object emitted by a run. See Foundations → Events for the full union.
CompactionSummarizing older messages to free context-window budget. Triggers automatically at 80% of the model's window.

Conventions for the implementer

  • TypeScript only, ESM, Bun 1.1+. No CommonJS dual builds. No Node runtime support — the SDK and CLI are Bun-native (see Project setup → Prerequisites).
  • No I/O in the core engine. The agent loop must not import from node:fs, node:child_process, bun:sqlite, or any logging library directly. Tools do I/O. The CLI does I/O. Sessions do I/O. The engine never does.
  • Errors are typed. Always throw a subclass of SkawldError — never a bare Error. See Foundations → Errors.
  • Events are exhaustive. The event union is closed; do not invent event types ad-hoc. New event kinds require a doc update first.
  • Every module exports a single public type or class. Tests live next to the source file (foo.ts + foo.test.ts).
  • Defer what is deferred. The doc set explicitly lists deferred features (subagents, MCP, WebFetch/Search, hooks, sandbox, file-state tracking, interactive CLI, default models). Task-state tools are included now only as persistent planning state; they do not spawn work.

Explicitly deferred (v1 does not ship these)

These appear in the architecture vocabulary so future development is easy, but they are not part of the v1 implementation.

  • Subagents. Persistent task-state tools are in v1, but no work is spawned or assigned to separate agents.
  • MCP server mode (exposing skawld's tools to other MCP clients), and MCP resources / prompts. The MCP client for third-party tools (stdio + Streamable HTTP) now ships — see the SDK doc.
  • WebFetch and WebSearch tools.
  • Hooks (pre/post tool, on-message, etc.).
  • Bash sandboxing.
  • File state tracking (mtime/hash detection between Read and Edit).
  • Interactive CLI (TUI, slash commands, live diffs).
  • A local tokenizer — usage comes from the API response.
  • A default model — the user always provides the model string explicitly.
