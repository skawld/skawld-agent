Permissions

Three coarse-grained modes, fine-grained allow/deny rules with path and command pattern matching, and a canUseTool callback as the final escape hatch. The engine is deterministic and decides per tool call.

The three modes

ModeReadsProject writes (Write/Edit)Task state (Task*)Exec (Bash)
"default" always allowed ask via canUseTool always allowed ask via canUseTool
"acceptEdits" always allowed auto-approved always allowed ask via canUseTool
"yolo" always allowed auto-approved always allowed auto-approved

The mode is the default decision. Rules can override the default in either direction — an explicit deny rule blocks something yolo would auto-approve; an explicit allow rule auto-approves something default would prompt for. Rules are evaluated before falling back to the mode default. Task-state tools are always allowed unless an explicit tool deny rule blocks them; they modify skawld session state, not project files.

Rule types

// src/permissions/rules.ts

export type PermissionRule =
  | ToolRule
  | PathRule
  | BashRule;

export interface ToolRule {
  kind: "tool";
  /** Tool name to match, or "*" for any. */
  tool: string;
  /** Decision used when matched. */
  decision: "allow" | "deny";
}

export interface PathRule {
  kind: "path";
  /** Tools this rule applies to. Default: project-file write tools (Write, Edit). */
  tools?: string[];
  /** Glob(s) the resolved absolute path must match. */
  paths: string[];
  decision: "allow" | "deny";
}

export interface BashRule {
  kind: "bash";
  /**
   * Pattern matched against the full command string.
   * - Plain string: prefix match after shell-tokenization (matches "git status", "git status -s", etc.)
   * - { regex: "..." }: tested as RegExp against the raw command string.
   */
  pattern: string | { regex: string };
  decision: "allow" | "deny";
}

Example rule sets

const rules: PermissionRule[] = [
  // Always allow git status and ls-style discovery.
  { kind: "bash", pattern: "git status",  decision: "allow" },
  { kind: "bash", pattern: "git diff",    decision: "allow" },
  { kind: "bash", pattern: "ls",          decision: "allow" },

  // Never run destructive commands without asking.
  { kind: "bash", pattern: { regex: "^\\s*(rm|sudo|chmod\\s+-R|chown\\s+-R)\\b" }, decision: "deny" },
  { kind: "bash", pattern: "git push",    decision: "deny" },

  // Never write outside the repo.
  { kind: "path", paths: ["!/Users/me/repo/**"], decision: "deny" },

  // Allow Edits inside src/ unconditionally.
  { kind: "path", tools: ["Edit"], paths: ["/Users/me/repo/src/**"], decision: "allow" },
];

Decision flow

For each pending tool call, the engine produces a decision in this order:

  1. Evaluate rules in array order. The first rule that matches wins. A rule matches when its kind, scope filters, and pattern all apply to the tool call.
  2. If no rule matched, apply the mode default (from the table above).
  3. If the result is "ask", invoke canUseTool. If no canUseTool was provided, treat "ask" as "deny" (fail-closed).
// src/permissions/engine.ts

export type PermissionDecision =
  | { decision: "allow" }
  | { decision: "deny"; reason: string }
  | { decision: "ask" };

export interface PendingToolCall {
  tool_use_id: string;
  tool: Tool;
  input: Record<string, unknown>;
}

export interface CanUseToolRequest {
  tool_name: string;
  tool_use_id: string;
  input: Record<string, unknown>;
  summary: string;
  /** The current mode (informational; the handler can ignore). */
  mode: PermissionMode;
}

export type CanUseToolResponse =
  | { behavior: "allow"; /** Optionally rewrite the input. */ updatedInput?: Record<string, unknown> }
  | { behavior: "deny";  message: string };

export type CanUseTool = (req: CanUseToolRequest, signal: AbortSignal) => Promise<CanUseToolResponse>;

export interface PermissionEngineOptions {
  mode: PermissionMode;
  rules: PermissionRule[];
  canUseTool?: CanUseTool;
  /** Absolute path the engine considers "the project root" for path-rule defaults. */
  projectRoot: string;
}

export class PermissionEngine {
  constructor(private opts: PermissionEngineOptions) {}

  /** Evaluate rules + mode only. Does NOT invoke canUseTool. */
  evaluate(call: PendingToolCall): PermissionDecision { /* ... */ }

  /**
   * Full resolution: evaluate(), then invoke canUseTool for "ask" results.
   * Returns a final allow/deny. Honors the abort signal.
   */
  async resolve(call: PendingToolCall, signal: AbortSignal): Promise<PermissionDecision> { /* ... */ }
}

Path rule matching

  • Paths are always resolved to absolute before matching. Relative inputs are resolved against ctx.cwd.
  • Matching uses picomatch (npm dep) with dot: true.
  • A leading ! in a pattern inverts the match (matches paths that do not match the rest). Useful for "deny anything outside this directory".
  • A PathRule with multiple patterns matches if any pattern matches.
  • Path-rule extraction per tool:
    • Read → file_path
    • Write → file_path
    • Edit → file_path
    • Bash → not applicable (use BashRule)
    • Glob → path if set, else ctx.cwd
    • Grep → path if set, else ctx.cwd
    • TaskCreate, TaskList, TaskGet, TaskUpdate → not applicable

Bash rule matching

  • Plain-string patterns match by token prefix: tokenize the command via a small shell-tokenizer (handle quotes), then require the rule's tokens to equal the command's first N tokens. "git status" matches git status, git status -s, and git status --short, but not git statusx or not-git status.
  • Regex patterns ({ regex: "..." }) are tested against the full untokenized command string with no implicit flags. Authors who want case-insensitive matching include (?i) or build the RegExp themselves at config time.
  • If the command contains shell operators (&&, ||, ;, |), the engine splits on them and evaluates each segment independently. The most restrictive decision among the segments wins (a single denied segment denies the whole command).
Treat composite commands as multiple commands

The model commonly composes shell with && and |. If the engine evaluates the whole string atomically, a single git status allow rule would not match git status && rm -rf node_modules, but the model could trivially evade rules by appending. Per-segment evaluation closes that escape.

canUseTool callback

The callback is the developer's chance to insert custom logic when rules + mode don't decide. It mirrors the Anthropic Agent SDK's canUseTool shape so the mental model carries over for users familiar with that ecosystem.

// Example: interactive permission prompt
const canUseTool: CanUseTool = async (req, signal) => {
  process.stdout.write(`\nTool: ${req.tool_name}\n${req.summary}\nApprove? [y/N] `);
  const answer = await readKeyPress(signal);
  return answer === "y"
    ? { behavior: "allow" }
    : { behavior: "deny", message: "user declined" };
};

Input rewriting

An allowed response may optionally rewrite the tool input via updatedInput. This is useful for redacting secrets, scoping paths, or transforming a destructive command into a dry-run. The engine substitutes the rewritten input before execution and reports it as the canonical input in the ToolCallStartEvent.

Abort behavior

The callback receives the run's AbortSignal. If the signal fires while the callback is awaiting user input, the callback should resolve with { behavior: "deny", message: "aborted" } or simply throw. The engine treats either as a deny.

Permission flow with parallel tool calls

When the model emits multiple tool calls in a single turn:

  1. Group calls by scope: reads[], then writes_and_execs[].
  2. For each group, evaluate rules + mode for every call. Collect the calls that resolved to "ask".
  3. Emit a single PermissionRequestEvent listing all "ask" calls in the group. Then invoke canUseTool once per call (mirroring the Anthropic Agent SDK: the callback decides per call; the consumer is free to batch their UI).
  4. For denied calls, synthesize a tool_result with is_error: true and the deny reason; do not execute.
  5. Execute approved calls per the parallel-execution rules in 02 — Tools.
// Sketch of the relevant loop fragment, for clarity.
const decisions = await Promise.all(pending.map(async (call) => {
  const r = engine.evaluate(call);
  if (r.decision !== "ask") return { call, decision: r };
  // Emit aggregated PermissionRequestEvent OUTSIDE this map; here we just resolve.
  const final = await engine.resolve(call, signal);
  return { call, decision: final };
}));

Where rules live

  • Inline in the Agent constructor (permissions: { mode, rules, canUseTool }).
  • In the project config file (.skawld/config.json, see 09 — Config).
  • In the user config file (~/.skawld/config.json).

When sources combine: user rules first, then project rules, then SDK-provided rules (most specific last). Within the combined array, first-match-wins still applies.

Putting it together

import { Agent, defaultTools, AnthropicProvider } from "skawld";
import type { CanUseTool, PermissionRule } from "skawld/permissions";

const rules: PermissionRule[] = [
  // Read-only safe commands: always allow.
  { kind: "bash", pattern: "git status", decision: "allow" },
  { kind: "bash", pattern: "git diff",   decision: "allow" },
  { kind: "bash", pattern: "ls",         decision: "allow" },
  { kind: "bash", pattern: "cat",        decision: "allow" },

  // Hard denies.
  { kind: "bash", pattern: { regex: "\\brm\\b" }, decision: "deny" },
  { kind: "bash", pattern: "git push",            decision: "deny" },

  // Scope writes to the repo only.
  { kind: "path", paths: ["!/Users/me/repo/**"], decision: "deny" },
];

const canUseTool: CanUseTool = async (req) => {
  console.log(`[ask] ${req.tool_name}: ${req.summary}`);
  // In a real app, prompt the user; here we deny.
  return { behavior: "deny", message: "no interactive prompt available" };
};

const agent = new Agent({
  provider: new AnthropicProvider({ apiKey: process.env.ANTHROPIC_API_KEY }),
  model: "claude-sonnet-4-6",
  tools: defaultTools(),
  permissions: { mode: "default", rules, canUseTool },
  cwd: "/Users/me/repo",
});

Audit log

Every decision the engine makes (rule match or callback result) is included as the summary field of the resulting ToolCallStartEvent for denied calls (which surface as tool_result is_error blocks). For approved calls, no special event is emitted — the normal ToolCallStartEvent is sufficient for audit. Consumers wanting a richer audit trail subscribe to the event stream and persist whichever events they care about.

Build checklist

  • src/permissions/rules.ts with PermissionRule union and matchers (path glob via picomatch, bash tokenizer for prefix match).
  • src/permissions/engine.ts with PermissionEngine, PermissionDecision, CanUseTool, CanUseToolRequest, CanUseToolResponse.
  • Composite-command splitting on && || ; | for bash rules; most-restrictive wins.
  • Rule precedence: first-match-wins within the combined array (user → project → SDK).
  • Fail-closed behavior when canUseTool is missing.
  • Optional input rewriting via updatedInput in the callback response.
  • Unit tests for: mode defaults per scope; rule precedence; path rule with absolute resolution; bash prefix match; bash regex match; composite command denial; canUseTool invocation and abort honor.
  • src/permissions/index.ts exporting all public types and the engine.
