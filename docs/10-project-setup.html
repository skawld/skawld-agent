Project setup

Everything needed to scaffold the repository: package.json with the right exports map, tsconfig.json, bun build for JS bundling, tsc for declaration emit, bun test as the test runner, and the src/ directory layout. Do this first so every subsequent module has a place to land.

Prerequisites

  • Bun 1.1+. skawld is Bun-native: Bun is the runtime, package manager, bundler, and test runner. The CLI ships with a #!/usr/bin/env bun shebang and the SQLite session store uses the built-in bun:sqlite module. Install from bun.sh.
  • Optional but recommended: ripgrep on PATH so Glob and Grep use the fast path instead of the JS fallback during development.
Why Bun-only

v1 commits to a single runtime. Bun gives us a built-in test runner (bun test), a built-in bundler (bun build), a native SQLite (bun:sqlite — no better-sqlite3 rebuild step), and substantially faster install and startup. The cost is that the SDK and CLI run under Bun only, not Node. Node compatibility is not a v1 goal.

Dependency notes

No better-sqlite3, no @types/node, no tsup, no vitest. SQLite ships with Bun (bun:sqlite); @types/bun covers the Bun globals plus the Node-compat surface; bun build replaces tsup; bun test replaces vitest. tsc stays — but only to emit .d.ts declarations, not JavaScript.

tsconfig.json

{
  "compilerOptions": {
    "target": "ES2022",

    "moduleResolution": "Bundler",
    "lib": ["ES2023"],
    "outDir": "dist",
    "rootDir": "src",
    "types": ["bun"],

    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "noPropertyAccessFromIndexSignature": false,

    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "emitDeclarationOnly": true,

    "esModuleInterop": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true,
    "allowImportingTsExtensions": false,

    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  },
  "include": ["src/**/*.ts"],
  "exclude": ["**/*.test.ts", "dist", "node_modules"]
}

Notes:

  • moduleResolution: "Bundler" matches how bun build and bun test resolve modules — extensionless imports and conditional exports resolve correctly.
  • emitDeclarationOnly: true — TypeScript emits only .d.ts files. JavaScript bundling is owned by bun build.
  • types: ["bun"] pulls in Bun globals plus the Node-compat surface (process, node:fs, etc.) via @types/bun.

Build: bun build + tsc

There is no separate build config file. The build is two commands wired up in scripts:

  1. bun build bundles each of the six entry points to dist/. The --root src flag preserves the subpath layout so src/providers/index.ts lands at dist/providers/index.js. --target bun emits Bun-flavored output. --sourcemap=linked writes external .js.map files.
  2. tsc --emitDeclarationOnly emits the matching .d.ts files into dist/.

The shebang at the top of src/cli/bin.ts survives the bundle; chmod 755 dist/cli/bin.js at the end of the build script makes it executable.

Test runner: bun test

No config file required. bun test picks up **/*.test.ts under the repo root by default. Tests import the runner API from bun:test — same surface as vitest (describe, it, expect):

// src/core/types.test.ts
import { describe, expect, it } from "bun:test";

describe("addUsage", () => {
  it("sums fields", () => {
    expect(1 + 1).toBe(2);
  });
});

Coverage is built in: bun test --coverage. For CI matrix testing, use the official oven-sh/setup-bun action — see the CI section below.

CLI bin shebang

#!/usr/bin/env bun
// src/cli/bin.ts
import { main } from "./main.js";
main(process.argv.slice(2)).catch((err) => {
  process.stderr.write(`skawld: ${err?.stack ?? err}\n`);
  process.exit(1);
});

Installed users invoke skawld directly; the shebang dispatches to their local bun. Bun must be on PATH for installed CLI usage — this is the runtime requirement.

Testing conventions

  • Unit tests next to source. src/tools/edit.ts ↔ src/tools/edit.test.ts. Tests do not import private internals via path tricks; if it cannot be tested through the module's exports, it probably belongs in its own module.
  • Mock provider in tests/helpers/mock-provider.ts. Implements BaseProvider with a scripted response sequence. The loop tests use it exclusively; never hit a real API in CI.
  • Filesystem isolation for tool tests: use os.tmpdir() + crypto.randomUUID() for per-test fixture directories; clean up in afterEach.
  • No network in unit tests. The Anthropic and OpenAI SDKs accept a custom fetch implementation; pass a fake in their tests. Bun's fetch is the global, so the mock simply assigns to globalThis.fetch when needed.
  • End-to-end CLI tests live in tests/e2e/ and spawn the built binary with a mock provider injected via env var. Use Bun.spawn for subprocess control.
  • Snapshot judgement: avoid snapshot tests for anything that includes ids, timestamps, or paths. Snapshots are OK for the NDJSON wire format after normalizing volatile fields.

Linting and formatting

  • v1 does not ship an ESLint or Prettier config. tsc --noEmit enforces correctness; team formatting conventions live in editor config. Add lint tooling later if the contributor base outgrows informal review.

CI

Add a GitHub Actions workflow that runs on PRs and pushes to main:

# .github/workflows/ci.yml
name: ci
on: [push, pull_request]
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        bun: [latest]
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
        with: { bun-version: ${{ matrix.bun }} }
      - run: bun install --frozen-lockfile
      - run: bun run typecheck
      - run: bun run build
      - run: bun test

Pin a specific Bun version (e.g. 1.1.34) once one is selected for the release; latest is fine while the project is pre-1.0.

Release flow

  • v1 ships from main. No separate release branch.
  • bun pm version <bump> → git push --follow-tags → bun publish --access public.
  • prepublishOnly runs build && test as a safety net.
  • A GitHub Actions workflow may automate this on tag push later; out of scope for v1.

Build checklist

Status: scaffold complete; CI and README still open.

  • package.json with the exact exports map and bin entry above; "engines": { "bun": ">=1.1" }.
  • tsconfig.json with "strict": true, the listed strictness flags, moduleResolution: "Bundler", and emitDeclarationOnly: true.
  • scripts.build:js invokes bun build with all six entry points, --root src, --target bun, --sourcemap=linked; scripts.build:types invokes tsc --emitDeclarationOnly; chmod 755 dist/cli/bin.js at the end.
  • .gitignore excluding node_modules, dist, .skawld, etc. (bun.lock is checked in.)
  • Empty stubs of every src/ file the layout calls for, each with a one-line placeholder comment so subsequent modules have somewhere to land. The build must succeed (no type errors) on this empty scaffold before module work starts.
  • Smoke test: bun install, bun run typecheck, bun run build, bun test all succeed.
  • GitHub Actions CI matrix (Linux/macOS/Windows × Bun latest) using oven-sh/setup-bun.
  • README with install + minimal examples + link to docs/index.html.
  • MIT LICENSE file.
