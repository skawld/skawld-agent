Project setup

Everything needed to scaffold the repository: package.json with the right exports map, tsconfig.json, tsup build, vitest test runner, and the src/ directory layout. Do this first so every subsequent module has a place to land.

Prerequisites

  • Node.js 20+. The package targets ESM with "type": "module" and uses node: built-ins.
  • npm 10+ (or pnpm / yarn — npm is documented here for simplicity).
  • Optional but recommended: ripgrep on PATH so Glob and Grep use the fast path instead of the JS fallback during development.

Dependency notes

better-sqlite3 is a native module and needs a prebuilt binary for the user's platform. It ships prebuilts for common targets. If a user installs on a platform without a prebuilt, npm will rebuild from source (requires a C++ toolchain). Document this in the README.

Testing conventions

  • Unit tests next to source. src/tools/edit.ts ↔ src/tools/edit.test.ts. Tests do not import private internals via path tricks; if it cannot be tested through the module's exports, it probably belongs in its own module.
  • Mock provider in tests/helpers/mock-provider.ts. Implements BaseProvider with a scripted response sequence. The loop tests use it exclusively; never hit a real API in CI.
  • Filesystem isolation for tool tests: use os.tmpdir() + crypto.randomUUID() for per-test fixture directories; clean up in afterEach.
  • No network in unit tests. The Anthropic and OpenAI SDKs accept a custom fetch implementation; pass a fake in their tests.
  • End-to-end CLI tests live in tests/e2e/ and spawn the built binary with a mock provider injected via env var.
  • Snapshot judgement: avoid snapshot tests for anything that includes ids, timestamps, or paths. Snapshots are OK for the NDJSON wire format after normalizing volatile fields.

Linting and formatting

  • ESLint with @typescript-eslint. A single flat eslint.config.js. Enforce: no any (with a few documented escape hatches), no unused vars, consistent imports.
  • Prettier for formatting. Default config except "printWidth": 100.
  • Pre-commit hook is not enforced in v1; trust CI.

CI

Add a GitHub Actions workflow that runs on PRs and pushes to main:

# .github/workflows/ci.yml
name: ci
on: [push, pull_request]
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node: [20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: ${{ matrix.node }} }
      - run: npm ci
      - run: npm run typecheck
      - run: npm run lint
      - run: npm run build
      - run: npm test

Release flow

  • v1 ships from main. No separate release branch.
  • npm version <bump> → git push --follow-tags → npm publish (with --access public).
  • prepublishOnly runs clean && build && test as a safety net.
  • A GitHub Actions workflow may automate this on tag push later; out of scope for v1.

Build checklist

  • package.json with the exact exports map and bin entry above.
  • tsconfig.json with "strict": true and the listed strictness flags; ESM-only with NodeNext.
  • tsup.config.ts producing the five subpath outputs + the CLI bin; chmod 0o755 on the bin file post-build.
  • vitest.config.ts picking up **/*.test.ts under src/ and tests/.
  • .gitignore excluding node_modules, dist, .skawld, etc.
  • Empty stubs of every src/ file the layout calls for, each with a one-line placeholder comment so subsequent modules have somewhere to land. The build must succeed (no type errors) on this empty scaffold before module work starts.
  • Smoke test: npm run build succeeds, npm test runs (passes with the zero placeholder tests).
  • GitHub Actions CI matrix (Linux/macOS/Windows × Node 20/22).
  • README with install + minimal examples + link to docs/index.html.
  • MIT LICENSE file.
