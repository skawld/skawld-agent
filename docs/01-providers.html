Providers

The LLM abstraction. BaseProvider defines the contract; AnthropicProvider, OpenAIChatCompletionsProvider, and OpenAIResponsesProvider implement it. The base classes are designed to be subclassed for compatible endpoints (Bedrock, Vertex, Groq, Ollama, vLLM, etc.) without modification to the engine.

The provider contract

A provider's job is narrow: take a normalized request (system prompt, tool schemas, messages, options), call the wire API, and yield a normalized stream of events that the engine turns into AssistantEvent, PartialAssistantEvent, and UsageEvent. Providers do not know about tools running, sessions, permissions, or compaction.

// src/providers/base.ts
import type {
  Message, ContentBlock, ModelId, Usage, StopReason,
} from "../core/types";
import type { ToolSchema } from "../tools/base";

/** A normalized request the engine builds for the provider on each turn. */
export interface ProviderRequest {
  model: ModelId;
  /** Already-assembled system prompt blocks (see 06-system-prompt). */
  system: SystemBlock[];
  /** Tool definitions in skawld's normalized shape. */
  tools: ToolSchema[];
  /** Full conversation so far. */
  messages: Message[];
  /** Hard cap on the response output tokens. */
  max_output_tokens: number;
  /** Optional temperature; provider chooses a sane default if omitted. */
  temperature?: number;
  /** Optional stop sequences. */
  stop_sequences?: string[];
  /** Hint to the provider whether to attempt prompt caching on this request. */
  cache_prompt?: boolean;
  /** Cache entry TTL hint. Default "5m"; "1h" for long-idle sessions. Anthropic only. */
  cache_ttl?: "5m" | "1h";
  /** Extended-thinking config. Anthropic only; other providers ignore it. */
  thinking?: ThinkingConfig;
  /** Effort hint (maps to Anthropic output_config.effort). Anthropic only. */
  effort?: EffortLevel;
  /** Abort signal for the whole request. */
  signal: AbortSignal;
}

export type EffortLevel = "low" | "medium" | "high" | "xhigh" | "max";

export type ThinkingConfig =
  | { type: "enabled"; budget_tokens: number; display?: "summarized" | "omitted" }
  | { type: "disabled" }
  | { type: "adaptive"; display?: "summarized" | "omitted" };

export interface SystemBlock {
  type: "text";
  text: string;
  /** True if this block should sit inside a cacheable prefix. */
  cacheable?: boolean;
}

/** Streaming events the provider yields. The engine assembles these into ContentBlocks. */
export type ProviderStreamEvent =
  | { type: "message_start"; model: ModelId }
  | { type: "text_delta"; text: string }
  | { type: "thinking_delta"; text: string; signature?: string }
  | { type: "tool_use_start"; id: string; name: string }
  | { type: "tool_use_input_delta"; id: string; json_delta: string }
  | { type: "tool_use_end"; id: string }
  | { type: "message_end"; stop_reason: StopReason; usage: Usage };

/** The final response after stream completion, materialized by the engine. */
export interface ProviderResponse {
  message: Message;             // role = "assistant"
  stop_reason: StopReason;
  usage: Usage;
}

export abstract class BaseProvider {
  /** Stable id for logs and config (e.g. "anthropic", "openai-chat"). */
  abstract readonly id: string;

  /** Maximum context window for the model, in tokens. Used by compaction. */
  abstract contextWindow(model: ModelId): number;

  /** Streaming entry point. The engine always streams. */
  abstract stream(req: ProviderRequest): AsyncIterable<ProviderStreamEvent>;
}
Why stream-only

The engine always wants partial events for the PartialAssistantEvent stream and to support early-cancel via AbortSignal. A non-streaming convenience method is unnecessary — consumers who want a final-only response can iterate the stream and keep only the last event.

Tool format translation

skawld's normalized tool shape is provider-agnostic:

// src/tools/base.ts (referenced here; full definition in 02-tools)
export interface ToolSchema {
  name: string;
  description: string;
  input_schema: {
    type: "object";
    properties: Record<string, JSONSchema>;
    required?: string[];
  };
}

Each provider translates this to its own wire format:

ProviderTool wire shape
Anthropic { name, description, input_schema } — already in our shape, no translation needed.
OpenAI Chat Completions { type: "function", function: { name, description, parameters: input_schema } }
OpenAI Responses { type: "function", name, description, parameters: input_schema } (function tools at top level, not nested)

Message format translation

The non-trivial part. skawld's normalized message has content: ContentBlock[]. Anthropic accepts this directly. OpenAI's two APIs each have their own conventions.

Anthropic

Direct mapping. tool_use and tool_result blocks are first-class. thinking blocks ship as-is with their signature. Image blocks ship as image source objects. The only translation is wrapping the system prompt blocks with cache_control: { type: "ephemeral" } on the blocks marked cacheable: true.

OpenAI Chat Completions

Each block becomes (part of) a separate message:

  • text in an assistant turn → { role: "assistant", content: "..." }.
  • tool_use blocks → { role: "assistant", tool_calls: [{ id, type: "function", function: { name, arguments: JSON.stringify(input) } }, ...] }. All tool_use blocks from one assistant turn go into the same message's tool_calls array.
  • tool_result blocks → one { role: "tool", tool_call_id, content } message per result.
  • thinking blocks → drop on the wire (OpenAI Chat Completions does not accept thinking on input).
  • image blocks → { role: "user", content: [{ type: "image_url", image_url: { url } }, ...] } (data URLs for base64).
  • System blocks → concatenate into a single { role: "system", content: "..." } at the top. The OpenAI prompt cache is automatic and prefix-based; no per-block annotation needed.

OpenAI Responses

Closer to skawld's shape because input items are typed. Map each block to a Responses input item:

  • text → { type: "message", role, content: [{ type: "input_text", text }] } (input) or output_text (output replay).
  • tool_use → { type: "function_call", call_id, name, arguments } on output replay.
  • tool_result → { type: "function_call_output", call_id, output }.
  • thinking → dropped on input unless the assistant message carries raw OpenAI Responses replay metadata; reasoning summaries are display content, not replay state.
  • image → { type: "input_image", image_url }.
  • System → concatenated into the top-level instructions field. Prompt caching is automatic.
Responses continuation

For the Responses API, skawld uses previous_response_id by default after the first OpenAI Responses turn. The provider stores the returned response id in namespaced message metadata and sends only new input items after that point. Stateless full-history mode remains available by disabling response chaining or setting store: false; in that mode, raw Responses output items are replayed and include: ["reasoning.encrypted_content"] is used when reasoning replay is needed.

Streaming

Each provider's stream() consumes its wire SSE stream and yields normalized ProviderStreamEvents in this order:

  1. message_start — exactly once, at the beginning.
  2. Zero or more deltas. text_delta and thinking_delta may interleave with tool-use delta sequences. A tool use is bracketed by tool_use_start → zero or more tool_use_input_delta → tool_use_end.
  3. message_end — exactly once, with the final stop_reason and usage.

If the stream is aborted, the iterator stops; the provider must clean up its underlying HTTP connection in a finally block. If the connection drops mid-stream, the provider throws a ProviderError with retryable: true.

Prompt caching

Anthropic

Explicit. The provider sets cache_control: { type: "ephemeral" } on at most two positions:

  • The last system block marked cacheable: true. A single breakpoint at the end of the cacheable run caches the entire tools + system prefix (the cache prefix hierarchy is tools → system → messages, so tools come "before" system in the cached prefix even though they sit in a separate request field). Per-block breakpoints would only help if individual system blocks had different stability profiles; ours don't.
  • When cache_prompt is true: one rolling breakpoint on the last content block of the most recent user message. A single rolling marker is intentional — multiple message-level markers cause Anthropic's serving tier to retain KV-cache pages that will never be read from, wasting cache pressure for no hit-rate gain. Thinking and tool_use blocks are skipped (they don't accept cache_control).

No breakpoint is placed on the tools array — the system breakpoint already caches the tools prefix as part of the tools → system chain. Total budget: 2 breakpoints used, 2 in reserve, well under Anthropic's 4-breakpoint cap. The cap is enforced as a defensive guard via MAX_CACHE_BREAKPOINTS but never bites in practice.

TTL

Every cache_control the provider emits respects the request's cache_ttl hint. Default ("5m" or omitted) emits a bare { type: "ephemeral" } — Anthropic's 5-minute ephemeral cache. Setting cache_ttl: "1h" emits { type: "ephemeral", ttl: "1h" } on every breakpoint, extending the entry TTL to 1 hour at 2x base cache-write cost. Use when gaps between turns may exceed 5 minutes (long-running batch jobs, idle interactive sessions). All breakpoints in a request share the same TTL — there is no per-breakpoint TTL knob. Note: flipping the TTL mid-session invalidates the existing cache entry (TTL is part of the cache key), so set it at Agent construction and leave it alone.

OpenAI (both APIs)

Automatic, server-side, prefix-based. Cache hits happen when the request's prefix matches a recent one. The provider does not annotate; it just keeps the request's prefix byte-stable (system prompt, then tools, then messages in order). Usage cache fields come back on the response and get reported as cache_read_tokens.

Compatible endpoints

Compatible OpenAI endpoints (Ollama, vLLM, Groq, etc.) generally do not return cache usage. Treat missing cache fields as zero. Do not error.

AnthropicProvider

// src/providers/anthropic.ts
import Anthropic from "@anthropic-ai/sdk";

export interface AnthropicProviderOptions {
  apiKey?: string;                    // falls back to ANTHROPIC_API_KEY
  baseURL?: string;                   // for proxies, Anthropic-compatible endpoints
  defaultHeaders?: Record<string, string>;
  maxRetries?: number;                // default 5; passed to retry helper
  thinking?: ThinkingConfig;          // default extended-thinking config (per-run overridable)
  effort?: EffortLevel;               // default effort hint (per-run overridable)
}

export class AnthropicProvider extends BaseProvider {
  readonly id = "anthropic";
  protected client: Anthropic;

  constructor(opts: AnthropicProviderOptions = {}) {
    super();
    this.client = new Anthropic({
      apiKey: opts.apiKey,
      baseURL: opts.baseURL,
      defaultHeaders: opts.defaultHeaders,
      maxRetries: 0,                  // we handle retries ourselves
    });
  }

  contextWindow(model: ModelId): number {
    // Static map of known models; default to 200_000 if unknown.
    return KNOWN_ANTHROPIC_CONTEXT[model] ?? 200_000;
  }

  async *stream(req: ProviderRequest): AsyncIterable<ProviderStreamEvent> {
    // 1. Translate req.system into Anthropic system blocks with cache_control.
    // 2. Translate req.messages (already in our format — minor field name shifts).
    // 3. Translate req.tools (already in our format).
    // 4. Open the stream with this.client.messages.stream(...).
    // 5. Map provider SSE events to ProviderStreamEvent and yield them.
    // 6. On message_stop, yield message_end with normalized stop_reason and usage.
  }
}

Extended thinking & effort

Two optional knobs control reasoning depth and token spend. Each can be set as a provider-level default in AnthropicProviderOptions and overridden per run via RunOptions.thinking / RunOptions.effort — the per-run value wins (req.thinking ?? this.thinking). They map onto the wire body verbatim:

  • thinking → request field thinking. Shapes: { type: "enabled", budget_tokens, display? }, { type: "disabled" }, or { type: "adaptive", display? }; display is "summarized" or "omitted".
  • effort → request field output_config.effort (note the nesting). Levels: "low" | "medium" | "high" | "xhigh" | "max"; "high" is the API default and equivalent to omitting.

skawld does not validate these client-side — combinations the API rejects (e.g. manual thinking on Opus 4.7, budget_tokens ≥ max_tokens, deprecated budget_tokens on 4.6/Sonnet 4.6) surface as a mapped 400 via mapAnthropicError. The interleaved-thinking beta header that some older models require (anthropic-beta: interleaved-thinking-2025-05-14) is set through the existing defaultHeaders option; there is no dedicated flag.

Caching interaction: the cache-breakpoint logic is unchanged by these knobs. cache_control is never placed on a thinking block (those don't accept it; applyConversationCacheBreakpoint skips them), and the system/tools cache prefix survives thinking-parameter changes. However, per Anthropic, changing thinking.type or budget_tokens between turns invalidates the conversation (message) cache breakpoint — so varying thinking/effort per run trades cache hits for flexibility. Set them once at construction when you want stable message-cache reuse.

Stop reason mapping:

"end_turn"   → "end_turn"
"tool_use"   → "tool_use"
"max_tokens" → "max_tokens"
"stop_sequence" → "stop_sequence"
"refusal"    → "refusal"
(other)      → "error"

OpenAIChatCompletionsProvider

// src/providers/openai-chat.ts
import OpenAI from "openai";

export interface OpenAIChatProviderOptions {
  apiKey?: string;                    // falls back to OPENAI_API_KEY
  baseURL?: string;                   // for Ollama/vLLM/Groq/DeepSeek/etc.
  defaultHeaders?: Record<string, string>;
  maxRetries?: number;
  /** Override context window lookup, for compatible endpoints. */
  contextWindowOverride?: (model: ModelId) => number | undefined;
}

export class OpenAIChatCompletionsProvider extends BaseProvider {
  readonly id = "openai-chat";
  protected client: OpenAI;
  protected contextWindowOverride?: (model: ModelId) => number | undefined;

  constructor(opts: OpenAIChatProviderOptions = {}) {
    super();
    this.client = new OpenAI({
      apiKey: opts.apiKey,
      baseURL: opts.baseURL,
      defaultHeaders: opts.defaultHeaders,
      maxRetries: 0,
    });
    this.contextWindowOverride = opts.contextWindowOverride;
  }

  contextWindow(model: ModelId): number {
    return this.contextWindowOverride?.(model) ?? KNOWN_OPENAI_CONTEXT[model] ?? 128_000;
  }

  async *stream(req: ProviderRequest): AsyncIterable<ProviderStreamEvent> {
    // 1. Concatenate system blocks into a single "system" message at the top.
    // 2. Translate messages: see "OpenAI Chat Completions" mapping above.
    // 3. Translate tools to { type: "function", function: { name, description, parameters } }.
    // 4. Open this.client.chat.completions.stream(...).
    // 5. Map provider delta events to ProviderStreamEvent.
    //    - choices[0].delta.content    → text_delta
    //    - choices[0].delta.tool_calls → tool_use_start / tool_use_input_delta / tool_use_end
    //    - finish_reason at the end    → stop_reason
    //    - usage from final chunk      → Usage (with cache fields if present)
  }
}

Stop reason mapping:

"stop"          → "end_turn"
"tool_calls"    → "tool_use"
"length"        → "max_tokens"
"content_filter" → "refusal"
(other)         → "error"
Subclass for compatible endpoints

OpenAI-compatible providers (Ollama, vLLM, Groq, DeepSeek, LM Studio, etc.) are not separate classes in v1. Users construct an OpenAIChatCompletionsProvider with the appropriate baseURL and contextWindowOverride. The base class is designed so that future first-party support (e.g. dedicated GroqProvider) is just a subclass that fixes the baseURL and provides a model→context map.

OpenAIResponsesProvider

// src/providers/openai-responses.ts

export interface OpenAIResponsesProviderOptions extends OpenAIChatProviderOptions {
  /** Reasoning effort hint or config, maps to the Responses API reasoning field. */
  reasoning?: OpenAIResponsesReasoningOption;
  /** Set false for stateless Responses requests. */
  store?: boolean;
}

export type OpenAIReasoningEffort =
  | "none" | "minimal" | "low" | "medium" | "high" | "xhigh";

export type OpenAIReasoningSummary = "auto" | "concise" | "detailed";

export type OpenAIResponsesReasoningOption =
  | OpenAIReasoningEffort
  | {
      effort?: OpenAIReasoningEffort;
      summary?: OpenAIReasoningSummary;
      previousResponseId?: "auto" | "disabled";
      encryptedContent?: boolean;
    };

export class OpenAIResponsesProvider extends BaseProvider {
  readonly id = "openai-responses";
  protected client: OpenAI;
  protected reasoning?: OpenAIResponsesReasoningOption;

  constructor(opts: OpenAIResponsesProviderOptions = {}) { /* ... */ }

  contextWindow(model: ModelId): number {
    return KNOWN_OPENAI_RESPONSES_CONTEXT[model] ?? 128_000;
  }

  async *stream(req: ProviderRequest): AsyncIterable<ProviderStreamEvent> {
    // 1. Concatenate system blocks into "instructions".
    // 2. Use previous_response_id when prior OpenAI response metadata exists.
    // 3. Otherwise translate messages into Responses input items
    //    (input_text / input_image / function_call / function_call_output / raw reasoning replay).
    // 4. Translate tools (function tools live at the top level).
    // 5. Open this.client.responses.stream(...).
    // 6. Map response.output_text.delta, response.function_call.* events,
    //    response.reasoning_summary_text.delta, and response.completed metadata.
  }
}

Reasoning summaries are opt-in through reasoning.summary. When OpenAI returns summary deltas, skawld maps them to thinking_delta and stores the assembled text as a ThinkingBlock so SDK consumers can display it. Encrypted reasoning content is stored only as provider replay metadata and is never surfaced as display text.

Stop / finish mapping mirrors Chat Completions; consult the Responses API's response.completed event for the analogue of finish_reason.

Image input

All three providers accept images in v1. The provider is responsible for translating ImageBlock to its wire format:

  • Anthropic: { type: "image", source: { type: "base64", media_type, data } | { type: "url", url } } inside a user message's content array.
  • OpenAI Chat: { type: "image_url", image_url: { url } }. For base64, build a data URL.
  • OpenAI Responses: { type: "input_image", image_url }. Same data-URL trick for base64.

Retry helper

// src/providers/retry.ts

export interface RetryOptions {
  maxAttempts: number;        // default 5
  baseDelayMs: number;        // default 1000
  maxDelayMs: number;         // default 30000
  jitter: number;             // 0..1, default 0.2
}

export interface StreamRetryOptions extends Omit<RetryOptions, "maxAttempts"> {
  maxRetries?: number;        // default 5; retries after first attempt
}

/**
 * Retry a function call that may throw a retryable SkawldError.
 * Honors RateLimitError.retry_after_seconds when present.
 */
export async function withRetry<T>(
  fn: (attempt: number) => Promise<T>,
  opts: RetryOptions,
  signal: AbortSignal,
): Promise<T> { /* ... */ }

export async function* withRetryableStream<T>(
  createStream: () => AsyncIterable<T>,
  opts: StreamRetryOptions,
  signal: AbortSignal,
): AsyncIterable<T> { /* ... */ }

The retry helpers wrap retryable SkawldErrors in the provider layer. Streaming providers use withRetryableStream: they disable SDK retries, map SDK errors into skawld errors, and retry only while no provider stream event has been yielded to the engine. Once any event has escaped, later stream failures do not retry because replaying the request could duplicate assistant or tool-use events. max_retries means retries after the first attempt; the default is 5 retries (6 total attempts).

Adding a new provider later

To add (for example) a GoogleProvider:

  1. Subclass BaseProvider in src/providers/google.ts.
  2. Implement contextWindow with a known-models map.
  3. Implement stream: translate request, open the wire stream, map events.
  4. Map errors per the table above.
  5. Export from src/providers/index.ts.

No engine change required.

Known model context windows

Ship a small static map in each provider file. These are starting points; users can override via contextWindowOverride (for OpenAI Chat) or by subclassing.

// Examples — keep updated as new models ship.
const KNOWN_ANTHROPIC_CONTEXT: Record<string, number> = {
  "claude-opus-4-6":   200_000,
  "claude-sonnet-4-6": 200_000,
  "claude-haiku-4-5":  200_000,
};

const KNOWN_OPENAI_CONTEXT: Record<string, number> = {
  "gpt-5":      400_000,
  "gpt-4.1":    1_000_000,
  "gpt-4o":     128_000,
  "o1":         200_000,
};
Don't hand-roll HTTP

Use the official @anthropic-ai/sdk and openai npm packages for SSE parsing and request signing. Disable SDK retries by passing maxRetries: 0, then apply skawld's retry helper around the mapped stream so Anthropic and OpenAI share one retry policy. Do not stack SDK and skawld retries: two retry layers multiply backoff delays.
