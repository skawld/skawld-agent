Config

JSON config files at the user and project levels, with a strict precedence chain. The CLI reads them; the SDK ignores them (developers configure the Agent in code). Schema is small, validated on load.

Files in this module

File locations

ScopePath
Project${cwd}/.skawld/config.json
User~/.skawld/config.json (resolved from os.homedir())

Both files are optional. Missing files are equivalent to {}. Malformed JSON throws ConfigError; schema violations throw ConfigError with a path-into-the-document.

Schema

// src/config/schema.ts
import type { PermissionMode } from "../core/types";
import type { PermissionRule } from "../permissions/rules";

export interface SkawldConfig {
  /** Provider id and per-provider options. */
  provider?: {
    id: "anthropic" | "openai-chat" | "openai-responses";
    apiKey?: string;
    baseURL?: string;
    /** Free-form per-provider options forwarded to the constructor. */
    options?: Record<string, unknown>;
  };

  /** Default model id. CLI flag --model overrides. */
  model?: string;

  /** Working directory. CLI flag --cwd overrides; default is process.cwd(). */
  cwd?: string;

  /** System prompt addendum. CLI flags override. */
  systemPrompt?: string;
  systemPromptFile?: string;

  /** Permissions: mode + rules. canUseTool is code-only, never in config. */
  permissions?: {
    mode?: PermissionMode;
    rules?: PermissionRule[];
  };

  /** Run-time limits. */
  limits?: {
    maxTurns?: number;
    maxOutputTokens?: number;
    maxRetries?: number;
  };

  /** Session store. v1 supports only "sqlite" and "memory" via config. */
  sessionStore?:
    | { kind: "sqlite"; databasePath?: string }
    | { kind: "memory" };

  /** Output options (CLI only). */
  output?: {
    partial?: boolean;
    pretty?: boolean;
  };
}

The loader validates against this shape. Unknown keys at the top level produce a warning to stderr (not an error) so future config additions don't break older CLI binaries. Unknown keys inside known objects (e.g. an unknown permissions field) produce a ConfigError — typos there matter.

Precedence

From lowest to highest. Higher entries win when keys overlap.

  1. Built-in defaults. Hard-coded values for things like provider fallback output caps (Anthropic uses 32768 when no maxOutputTokens is set; maxTurns defaults to unbounded).
  2. User config (~/.skawld/config.json) — applies across projects.
  3. Project config (${cwd}/.skawld/config.json) — applies to the project.
  4. Environment variables — see the table below.
  5. CLI flags — the final word.
Merge semantics, not replacement

Object-typed keys (like permissions, provider, limits) are shallow-merged. Array-typed keys (like permissions.rules) are concatenated in precedence order: user rules, then project rules, then CLI-supplied rules. First-match-wins (per 03 — Permissions) means earlier in the concatenated list wins. To force an override, put the override in the source closer to the top of the chain.

loadConfig API

// src/config/loader.ts

export interface LoadConfigOptions {
  cwd: string;
  /** Override the project config path. Default: ${cwd}/.skawld/config.json. */
  projectConfigPath?: string;
  /** Skip the user config file even if it exists. */
  ignoreUserConfig?: boolean;
  /** CLI overrides, already parsed into the SkawldConfig shape. */
  cliOverrides?: Partial<SkawldConfig>;
  /** Environment snapshot, default process.env. */
  env?: NodeJS.ProcessEnv;
}

export interface LoadedConfig {
  config: SkawldConfig;
  /** Sources contributing to the merged result, for debugging. */
  sources: Array<{ source: "default" | "user" | "project" | "env" | "cli"; path?: string }>;
}

export async function loadConfig(opts: LoadConfigOptions): Promise<LoadedConfig> { /* ... */ }

loadConfig is called by the CLI. The SDK does not call it — SDK consumers configure the Agent directly. If a developer wants to apply a project config file in code, they call loadConfig themselves and pass the result into new Agent({...}).

Example project config

{
  "provider": {
    "id": "anthropic"
  },
  "model": "claude-sonnet-4-6",
  "systemPromptFile": "./.skawld/instructions.md",
  "permissions": {
    "mode": "default",
    "rules": [
      { "kind": "bash", "pattern": "git status", "decision": "allow" },
      { "kind": "bash", "pattern": "git diff",   "decision": "allow" },
      { "kind": "bash", "pattern": "npm test",   "decision": "allow" },
      { "kind": "bash", "pattern": { "regex": "\\brm\\b" }, "decision": "deny" },
      { "kind": "bash", "pattern": "git push",   "decision": "deny" }
    ]
  },
  "limits": {
    "maxTurns": 200
  },
  "sessionStore": {
    "kind": "sqlite",
    "databasePath": "./.skawld/sessions.db"
  }
}

Example user config

{
  "provider": { "id": "anthropic" },
  "permissions": {
    "rules": [
      { "kind": "bash", "pattern": "ls",  "decision": "allow" },
      { "kind": "bash", "pattern": "pwd", "decision": "allow" },
      { "kind": "bash", "pattern": "cat", "decision": "allow" }
    ]
  },
  "output": { "partial": false }
}

Combined with the project config above, the merged permissions.rules array (user → project → CLI) is:

[ls, pwd, cat, git status, git diff, npm test, !rm, !git push]

Validation

  • Parse JSON with a try/catch; bad JSON → ConfigError with the file path and line/column from the parser.
  • Validate the shape with a small hand-rolled validator (the surface area is small — no need for ajv). For each known field, check type and recurse into nested objects. Report the JSON Pointer of the offending field.
  • Unknown top-level keys → log a warning, ignore. Unknown nested keys → error.
  • For permissions.rules, validate each rule per its kind.

Security notes

  • Never put secrets in the config file. Provider apiKey in config is honored but not recommended; prefer the env var. The CLI should warn (to stderr) when it sees an apiKey coming from a file on disk.
  • Permission rules in config are not authoritative for the SDK. SDK consumers always pass rules in code; config rules apply only when used via the CLI or when the SDK consumer explicitly opts in by calling loadConfig.
  • Project config in untrusted repos: if a checked-in .skawld/config.json tries to declare "yolo" mode or pre-allow dangerous commands, the CLI should refuse to apply it unless invoked with --accept-project-config. Treat the project config as untrusted by default; the user explicitly accepts it once.
Refuse untrusted permissive configs by default

An attacker-controlled .skawld/config.json committed to a repo could pre-allow rm -rf or set "yolo" mode. The loader must detect any rule whose decision is "allow" for a bash regex/pattern, or a mode of "acceptEdits"/"yolo", in the project file, and require an explicit --accept-project-config (or a one-time prompt on first encounter) before applying. User config is trusted; project config is not.

Build checklist

  • src/config/schema.ts with SkawldConfig and a hand-rolled validator that reports JSON Pointers.
  • src/config/loader.ts with loadConfig implementing precedence: defaults → user → project → env → CLI overrides.
  • Object keys shallow-merge; permissions.rules arrays concatenate in precedence order.
  • Env-var mapping per the table.
  • Unknown-key handling: warn at top level, error at nested level.
  • Untrusted-project-config gate requiring --accept-project-config for permissive rules / modes.
  • Tests: missing files OK; bad JSON errors; precedence correctness for every overlap; rule concatenation order; env-var coverage; warn vs error on unknown keys; security gate fires for permissive project configs.
