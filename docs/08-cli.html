docs / 08 — cli

CLI

A non-interactive binary that streams NDJSON events on stdout and accepts JSON control messages on stdin. The stdio protocol is the polyglot wire format — a future Python SDK will wrap it. Interactive TUI mode is a future milestone, not v1.

Files in this module

src/cli/main.tsArgument parsing, subcommand dispatch
src/cli/run.tsrun subcommand: build Agent, stream events
src/cli/sessions.tssessions subcommand: list / show / delete
src/cli/protocol.tsNDJSON serialization, stdin control parser

Install & invoke

# Local
bun add skawld
bunx skawld --help

# Global
bun add -g skawld
skawld --help

Bun must be on the user's PATH; the installed binary's shebang is #!/usr/bin/env bun.

Subcommands

skawld run [prompt]            Start or resume a session with a prompt.
skawld sessions list           List recent sessions.
skawld sessions show <id>      Print the message history for a session.
skawld sessions delete <id>    Delete a session and its messages.
skawld --version               Print the version.
skawld --help                  Print help.

  --system-prompt-file <path>  Read system prompt addendum from a file.

  --max-turns <n>              Default: unbounded (no cap).
  --max-output-tokens <n>      Default 8192.
  --max-retries <n>            Default 5.

OUTPUT
  --partial                    Include partial_assistant events.
  --no-cache                   Disable prompt caching hints to the provider.
  --pretty                     Pretty-print JSON for human reading (default: NDJSON).

CONFIG
  --config <path>              Path to a project config file. Default: ./.skawld/config.json.
  --no-user-config             Ignore ~/.skawld/config.json.

Stdout — NDJSON events

Every line on stdout is one JSON object — an Event from the union defined in 00 — Foundations. Order matches the order events are produced by the loop.

{"type":"system","subtype":"init","session_id":"01HW...","run_id":"01HW...","model":"claude-sonnet-4-6","tools":["Read","Write","Edit","Bash","Glob","Grep","TaskCreate","TaskList","TaskGet","TaskUpdate"],"permission_mode":"default","cwd":"/Users/me/repo"}
{"type":"user","message":{"role":"user","content":[{"type":"text","text":"<env>Today's date: 2026-05-18</env>"},{"type":"text","text":"fix the failing test"}]}}
{"type":"assistant","message":{"role":"assistant","content":[{"type":"text","text":"Let me look at the test file."},{"type":"tool_use","id":"toolu_01","name":"Read","input":{"file_path":"src/parse.test.ts"}}]},"stop_reason":"tool_use"}
{"type":"tool_call_start","tool_use_id":"toolu_01","tool_name":"Read","input":{"file_path":"src/parse.test.ts"}}
{"type":"tool_call_end","tool_use_id":"toolu_01","tool_name":"Read","is_error":false,"duration_ms":12}
{"type":"user","message":{"role":"user","content":[{"type":"tool_result","tool_use_id":"toolu_01","content":"   1\timport ...\n..."}]}}
{"type":"usage","usage":{"input_tokens":1832,"output_tokens":117,"cache_read_tokens":1500},"cumulative":{"input_tokens":1832,"output_tokens":117,"cache_read_tokens":1500}}
{"type":"result","subtype":"success","stop_reason":"end_turn","total_usage":{"input_tokens":5421,"output_tokens":342},"duration_ms":4218,"final_text":"Fixed: the comparator was reversed."}

With --pretty each event is pretty-printed JSON separated by blank lines instead of one event per line. NDJSON is the default because that is the polyglot-friendly format.

Skill events (planned — module 11)

When the Agent loads at least one skill from <configDir>/skills/ (see 11 — Skills), three additional event types appear on the public Event union. They are emitted today by the programmatic SDK; the NDJSON relay will be wired here when CLI module 08 ships.

{"type":"skills_loaded","skills":[{"name":"commit","description":"Stage changes and create a commit.","when_to_use":"When the user asks to commit changes.","argument_hint":"<message>"}]}
{"type":"skill_invoked","name":"commit","args":"fix bug"}
{"type":"skill_completed","name":"commit","is_error":false}

skills_loaded fires once per session immediately after system. skill_invoked / skill_completed bracket each Skill tool call (around the existing tool_call_start / tool_call_end pair).

Protocol integrity — keeping stdout JSON-only

The NDJSON contract is: every line on stdout is a valid Event. The OS does not enforce this — anything that writes to file descriptor 1 (FD 1) lands in the same byte stream as the protocol. A single stray console.log from a transitive dependency, a deprecation warning, or a startup banner will appear mid-stream and crash a strict line-by-line JSON.parse consumer.

To uphold the contract, the CLI must isolate stdout for protocol use only:

  • Redirect console.* to stderr at CLI entry. Before importing anything else, the binary replaces globalThis.console with a shim whose log, info, warn, error, debug, and trace methods all write to process.stderr. This catches engine code, third-party libs, and Bun's own info output.
  • Reserve process.stdout.write for the protocol writer. The NDJSON serializer in src/cli/protocol.ts is the only call site that writes to FD 1. Everything else — diagnostics, banners, progress, prompts — goes to FD 2.
  • Silence known-chatty environments. Set NODE_NO_WARNINGS=1 and suppress any Bun startup messages that target stdout. Audit dependencies for libraries that write directly to process.stdout (not just console.log) and avoid or wrap them.
  • Defense in depth on the consumer side. SDKs reading the pipe should still wrap JSON.parse in try/catch per line and route non-JSON lines to a log channel rather than throwing. The CLI's job is to never produce them; the SDK's job is to not crash if a future regression does.

This is the same reason the engine has zero I/O assumptions (see Overview → Architecture): the CLI is the one place that owns stdout, and it owns it exclusively.

Stdin — control messages

Stdin is line-oriented JSON. The CLI parses each line as one control message. Unrecognized message types are ignored.

type ControlMessage =
  | { type: "permission_response"; tool_use_id: string; decision: "allow" | "deny"; reason?: string; updated_input?: Record<string, unknown> }
  | { type: "abort"; reason?: string }
  | { type: "prompt"; text: string };          // for follow-up prompts in the SAME session

Permission flow over stdio

When --canusetool is set and the engine encounters an "ask" decision, the CLI:

  1. Emits a permission_request event on stdout (which contains one or more pending tool calls and their summaries).
  2. Suspends progress for those calls until a matching permission_response arrives on stdin for each tool_use_id.
  3. Once all responses arrive, the engine continues; denied calls become is_error results.
// stdout
{"type":"permission_request","requests":[{"tool_use_id":"toolu_07","tool_name":"Bash","input":{"command":"npm test"},"summary":"Run: npm test"}]}

// stdin (consumer writes this)
{"type":"permission_response","tool_use_id":"toolu_07","decision":"allow"}

Without --canusetool, "ask" decisions are treated as denies (fail-closed). This matches the SDK default when no canUseTool callback is provided.

Abort over stdio

{"type":"abort","reason":"user requested"}

Sending an abort message causes the run to terminate with a result event of subtype: "aborted" and a non-zero exit code (130). SIGINT (Ctrl-C) behaves identically when the CLI is run from a TTY.

Follow-up prompts

After a run completes, additional { "type": "prompt", "text": "..." } control messages add new runs to the same session. The CLI exits when stdin is closed (EOF) and no run is in flight. This is what lets a polyglot SDK keep a long-lived skawld process around for back-and-forth conversations.

# Programmatic multi-turn use:
#   echo '{"type":"prompt","text":"now refactor the parser"}' >> named-pipe
skawld run "explain the parser" --session abc123 < named-pipe > output.ndjson

Stderr

Stderr is for human-readable diagnostics:

  • Argument parsing errors.
  • Catastrophic startup failures (missing API key, invalid model).
  • Optional progress messages from the CLI itself (debug logging when SKAWLD_LOG=debug).

Stderr is not JSON. Anything machine-readable goes on stdout.

Exit codes

CodeMeaning
0Successful completion — final result event had subtype: "success".
1Operational error — final result event had subtype: "error".
2Usage error — bad arguments, missing required option, unknown subcommand.
3Configuration error — config file unreadable, malformed, or contains invalid rules.
4Authentication error — provider returned 401/403.
130Aborted — SIGINT or stdin abort control message.

skawld sessions

list

skawld sessions list [--limit 20] [--offset 0]

Emits one NDJSON line per session: { id, created_at, updated_at, meta, message_count }. With no flags, lists the 20 most recently updated sessions.

show

skawld sessions show <id>

Emits one NDJSON line per StoredMessage: { seq, appended_at, message }.

delete

skawld sessions delete <id> [--yes]

Deletes the session, its messages, and its persistent tasks. Prompts on stderr unless --yes. Exits 0 on success.

Examples

One-shot

skawld run "find all FIXME markers in src/ and summarize them" \
  --model claude-sonnet-4-6 \
  --permission-mode acceptEdits

Programmatic pipe

skawld run "rename foo to bar across the repo" \
  --model gpt-5 --provider openai-chat \
  --permission-mode default --canusetool \
  < permission-driver.json | grep '"type":"result"'

Resume

skawld sessions list --limit 5 | head -1
# {"id":"01HW...","created_at":"...","updated_at":"...","meta":{},"message_count":42}
skawld run "what was the last thing we discussed?" \
  --session 01HW... --model claude-sonnet-4-6

Multi-turn over stdin

cat <<EOF | skawld run --model claude-sonnet-4-6 --session demo
{"type":"prompt","text":"list the files in src/"}
{"type":"prompt","text":"now read src/main.ts"}
EOF

Signals

  • SIGINT (Ctrl-C): triggers abort. Exits 130.
  • SIGTERM: triggers abort. Exits 143.
  • SIGPIPE on stdout (consumer closed the pipe): triggers abort.

Implementation notes

  • Use commander or a tiny hand-rolled argument parser. Argument parsing must be deterministic and produce sane error messages.
  • Wire stdin via readline.createInterface({ input: process.stdin }) and dispatch on the line event.
  • Build the Agent inside run.ts from CLI args + loaded config (see 09 — Config).
  • The CLI is the only place in v1 that touches process.stdout, process.stderr, or process.stdin for protocol purposes. The engine never does.
  • At the very top of src/cli/bin.ts — before any other imports — replace globalThis.console with a shim that writes every method to process.stderr. This prevents transitive dependencies from polluting the NDJSON stream on FD 1. See Protocol integrity.
  • Only the NDJSON serializer in src/cli/protocol.ts may call process.stdout.write. Treat any other reference to process.stdout as a bug.
  • Pretty-print mode (--pretty) uses JSON.stringify(event, null, 2) followed by a blank line.
  • Permission-response correlation: keep a Map<tool_use_id, Promise>; resolve on incoming response.
No interactive TUI in v1

The CLI does not draw a TUI, does not handle slash commands, and does not show colored diffs. It is the boring NDJSON pipe. The future interactive CLI will be a separate frontend on the same engine — see Overview → Architecture.

Build checklist

  • src/cli/bin.ts with shebang #!/usr/bin/env bun calling main().
  • src/cli/main.ts with subcommand routing and global flags.
  • src/cli/run.ts implementing the run subcommand: load config, build Agent (provider switch on --provider), open session, stream events to stdout as NDJSON.
  • Stdin line reader dispatching control messages.
  • Permission-response correlation via a per-tool_use_id promise map.
  • Signal handling (SIGINT → abort, exit 130; SIGTERM → abort, exit 143; SIGPIPE → abort).
  • src/cli/sessions.ts implementing list, show, delete.
  • src/cli/protocol.ts with NDJSON serializer (one event per line, trailing newline) and stdin parser.
  • globalThis.console redirected to stderr in src/cli/bin.ts before any other import, with a test asserting that console.log never appears on stdout.
  • Exit codes per the table.
  • End-to-end tests using a mock provider: stdout NDJSON shape, stdin permission flow, abort over stdin, signals, exit codes.
  • package.json "bin" field pointing at dist/cli/bin.js.
