System prompt

A cache-optimized assembly. The system message holds only stable content. Dynamic content (date, per-run env) goes into the first user message. The goal: maximize prompt-cache hit rate across sessions, days, and runs.

Why cache shape matters

Both Anthropic and OpenAI cache prompts prefix-wise: the first N tokens of the request are reused from cache only when they are byte-identical to a recent request. A single shifting byte at the top invalidates the entire cached prefix. The system prompt is the largest stable thing skawld sends, so we have to keep it pure.

Concretely, this means:

  • Do not put the current date in the system prompt. The date changes daily; a session resumed tomorrow would miss the cache.
  • Do not put session ids, run ids, or timestamps in the system prompt. They are different every run.
  • Do put OS, cwd, agent version, and user custom instructions in the system prompt. They are stable per Agent instance.

Layout

The system prompt is built as an ordered array of SystemBlocks (see 01 — Providers). Providers turn this into wire format.

// src/core/system-prompt.ts

export interface SystemPromptInputs {
  /** User's appended custom instructions (Agent.opts.systemPrompt). */
  userInstructions?: string;
  /** Working directory. */
  cwd: string;
  /** OS info: derive at Agent construction from process.platform + os.release(). */
  os: { platform: string; release: string; arch: string };
  /** Shell, derived from process.env.SHELL or "unknown". */
  shell: string;
  /** Node version. */
  nodeVersion: string;
  /** skawld package version. */
  skawldVersion: string;
  /** Names of registered tools, in stable order. */
  toolNames: string[];
  /** Permission mode. */
  permissionMode: PermissionMode;
}

export function buildSystemBlocks(inp: SystemPromptInputs): SystemBlock[] {
  return [
    { type: "text", text: BLOCK_A_IDENTITY,                       cacheable: true },
    { type: "text", text: BLOCK_A_TOOL_PROTOCOL,                  cacheable: true },
    { type: "text", text: blockBEnvironment(inp),                 cacheable: true },
    ...(inp.userInstructions
        ? [{ type: "text", text: blockBUserInstructions(inp.userInstructions), cacheable: true }]
        : []),
  ];
}

Block A — identity (most stable)

export const BLOCK_A_IDENTITY = `
You are skawld, an autonomous software engineering agent. You work inside a
codebase on the user's computer, using the file and shell tools provided to
read, modify, and run code. Your goal is to complete the user's coding task
correctly and minimally.

Behave like an experienced engineer: read before you write, run before you
claim, prefer small focused changes, surface uncertainty instead of guessing.
Do not narrate trivial operations. Do not produce status reports unless the
user asked. When in doubt, ask one focused question.
`.trim();

Block A — tool-use protocol (most stable)

export const BLOCK_A_TOOL_PROTOCOL = `
Tool use protocol:

- Read a file before you Edit it. The Edit tool will refuse if you have not.
- Edits require an exact byte-for-byte match of the old_string in the current
  file contents, including indentation. The line-number prefix shown by Read
  is NOT part of the file content; do not include it in old_string.
- Prefer Edit over Write when modifying an existing file. Use Write only for
  new files or full rewrites.
- Glob is for finding files by name pattern; Grep is for searching file
  contents. They are read-only; they do not modify anything.
- Bash runs in the user's environment with full permissions. There is no
  sandbox. Avoid commands that change global state unless the user asked for
  them. Default timeout is 2 minutes; use timeout_ms for longer builds/tests up
  to 30 minutes.
- Issue multiple tool calls in a single turn when they are independent. Read,
  Glob, and Grep can run concurrently; the engine will parallelize them.
- Use TaskCreate, TaskList, TaskGet, and TaskUpdate to track multi-step work.
  Mark tasks in_progress before working on them and completed when finished.
  Use TaskList to refresh state before continuing long or resumed work.
`.trim();

Block B — environment

function blockBEnvironment(inp: SystemPromptInputs): string {
  return `
Environment:

- skawld version: ${inp.skawldVersion}
- Node: ${inp.nodeVersion}
- OS: ${inp.os.platform} ${inp.os.release} (${inp.os.arch})
- Shell: ${inp.shell}
- Working directory: ${inp.cwd}
- Permission mode: ${inp.permissionMode}
- Tools available: ${inp.toolNames.join(", ")}
`.trim();
}

Block B — user custom instructions (optional)

function blockBUserInstructions(text: string): string {
  return `
User-provided instructions:

${text.trim()}
`.trim();
}

Dynamic content: the env user prefix

Dynamic content goes into the first user message of every run, as a leading text block. This keeps it visible to the model without contaminating the cacheable prefix.

export function buildEnvUserPrefix(): string {
  return `
<env>
Today's date: ${new Date().toISOString().slice(0, 10)}
</env>
`.trim();
}

The Agent calls buildEnvUserPrefix() when constructing the user message for each run():

function buildUserMessage(prompt: string, images?: ImageInput[]): Message {
  const envBlock: TextBlock = { type: "text", text: buildEnvUserPrefix() };
  const promptBlock: TextBlock = { type: "text", text: prompt };
  const imageBlocks = (images ?? []).map(toImageBlock);
  return { role: "user", content: [envBlock, promptBlock, ...imageBlocks] };
}
Why a separate <env> block, not concatenated

Keeping the env in its own text block (rather than concatenating it with the prompt) makes it visually distinct to the model and easy to spot in logs. The XML-ish wrapper is a convention the major models handle well.

Caching annotation

Anthropic

The provider sets cache_control: { type: "ephemeral" } on every system block where cacheable: true. It also sets one breakpoint on the tools array (after the last tool). Total breakpoints used by the system prompt + tools: at most 3 or 4, comfortably under Anthropic's 4-breakpoint limit. Conversation-prefix caching (an additional breakpoint on the latest tool_result of a long run) is left to the loop.

OpenAI (Chat Completions and Responses)

No annotation needed. Cache is automatic and prefix-based. The provider simply ensures the system content's byte order is stable: Block A identity ‖ Block A tool protocol ‖ Block B env ‖ Block B user instructions (if any), then the tools array, then the messages. As long as the inputs to buildSystemBlocks do not change between runs (and they should not, for a given Agent instance), the prefix is identical and cache hits accumulate.

User custom instructions

The Agent accepts systemPrompt?: string which is appended as Block B. Stable per Agent. Not session-scoped — different sessions in the same Agent share the same custom instructions, which is what enables the cache prefix to stay stable.

If a consumer needs per-session instructions, they construct a new Agent per session. This is intentional: per-session custom instructions break the cache prefix, so the API forces the consumer to opt in by paying the construction cost.

Why this ordering matters

  1. Identity first — the most stable string in the whole prompt, the cheapest to keep cached forever.
  2. Tool protocol next — only changes when we ship a new version of skawld. Almost as stable.
  3. Environment — stable per Agent instance (per cwd, per OS, per tool set). The first block that varies between users.
  4. User instructions — last cacheable block, the most likely thing for a user to tweak between runs of similar Agents.
  5. Then tools, then messages. Tools are part of the request body, not the system message; the provider treats them as part of the cacheable prefix anyway.

What does NOT go in the system prompt

  • Date / time — goes in the env user prefix.
  • Session id, run id — informational only; surfaced via events, not via the prompt.
  • Git branch / commit info — also dynamic; if you want the model to know, put it in the env user prefix or a user message.
  • File contents — never. The model uses Read on demand.
  • Per-run options (temperature, etc.) — passed through the request, not the prompt.

Build checklist

  • src/core/system-prompt.ts with buildSystemBlocks, buildEnvUserPrefix, exported text constants for the two Block A pieces.
  • All cacheable blocks tagged with cacheable: true.
  • Block ordering: identity → tool protocol → env → user instructions.
  • No date/time/session-id/run-id in any system block.
  • Env block formats deterministically (sorted tool names, stable line order).
  • The Agent invokes buildSystemBlocks exactly once at construction and reuses the result for every run. (src/core/agent.ts — systemBlocks built in constructor, stored in agentInternals)
  • buildEnvUserPrefix invoked once per run, attached as the first text block of the first user message of that run. (src/core/loop.ts — buildUserMessage)
  • Tests: byte-identical output for identical inputs across multiple calls; cacheable flag set on all system blocks; user instructions block omitted when not provided.
