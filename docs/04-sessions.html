Sessions

A session is a persistent conversation identified by a stable id. The SessionStore interface defines how messages and task state are loaded, appended, updated, and listed. The default SqliteSessionStore ships with skawld; alternative stores (Postgres, Mongo, in-memory) implement the same interface.

The session model

Every time the SDK starts work, it is doing so inside a session. There is no "stateless one-shot" mode. The pattern is:

  • agent.session() with no id → creates a new session and returns a fresh Session object.
  • agent.session({ id }) with an existing id → loads that session's messages from the store and returns a Session bound to it.
  • agent.session({ id }) with an id that does not exist → creates a new session under that id (idempotent create). This is useful when the caller wants to mint ids themselves.

A session contains many runs. Each run is one user prompt → agent works → final result cycle. Runs append to the session's message history; ending a run does not close the session. A session also owns a persistent task list used by TaskCreate, TaskList, TaskGet, and TaskUpdate.

Task types

Tasks are session-scoped persistence records used by the task tools in 02 — Tools. They live in the sessions module because the session store is the persistence boundary.

// src/sessions/tasks.ts
export type TaskStatus = "pending" | "in_progress" | "completed";

export interface Task {
  id: string;
  session_id: string;
  subject: string;
  description: string;
  active_form?: string;
  status: TaskStatus;
  owner?: string;
  blocks: string[];
  blocked_by: string[];
  metadata?: Record<string, unknown>;

  description: string;
  active_form?: string;

}

SessionStore interface

// src/sessions/store.ts
import type { Message } from "../core/types";
import type { CreateTaskInput, Task, TaskPatch } from "./tasks";

/** Metadata stored about a session. */
export interface SessionRecord {
  id: string;
  created_at: string;     // ISO-8601
  updated_at: string;     // ISO-8601
  /** Arbitrary developer-supplied metadata: title, tags, project root, etc. */
  meta: Record<string, unknown>;
}

/** A message with append metadata. */
export interface StoredMessage {
  /** Monotonic sequence within the session, starting at 1. */
  seq: number;
  /** ISO-8601 timestamp when the message was appended. */
  appended_at: string;
  /** The message itself, in skawld's normalized shape. */
  message: Message;
}

export interface SessionStore {
  /** Create a session. If id is provided and already exists, return the existing record. */
  create(record: { id?: string; meta?: Record<string, unknown> }): Promise<SessionRecord>;

  /** Load metadata for a session. Returns undefined if not found. */
  load(id: string): Promise<SessionRecord | undefined>;

  /** Load all messages for a session in seq order. */
  loadMessages(id: string): Promise<StoredMessage[]>;

  /**
   * Append one or more messages atomically.
   * Implementations MUST assign monotonic seq values and update updated_at on the session.
   */
  appendMessages(id: string, messages: Message[]): Promise<StoredMessage[]>;

  /** Update the session's metadata (shallow merge). */
  updateMeta(id: string, meta: Record<string, unknown>): Promise<SessionRecord>;

  /** List sessions, most recently updated first. */
  list(opts?: { limit?: number; offset?: number }): Promise<SessionRecord[]>;

  /** Delete a session and all its messages, tasks, task edges, and task counters. */
  delete(id: string): Promise<void>;

  /** Create a persistent task in this session. Assigns a stable, monotonically increasing id. */
  createTask(sessionId: string, input: CreateTaskInput): Promise<Task>;

  /** Load one task. Returns undefined if the task does not exist. */
  getTask(sessionId: string, taskId: string): Promise<Task | undefined>;

  /** List all tasks for this session, sorted by numeric id ascending. */
  listTasks(sessionId: string): Promise<Task[]>;

  /** Patch one task. Returns undefined if the task does not exist. */
  updateTask(sessionId: string, taskId: string, patch: TaskPatch): Promise<Task | undefined>;

  /** Delete one task and its dependency edges. Returns whether a row was deleted. */
  deleteTask(sessionId: string, taskId: string): Promise<boolean>;

  /**
   * Release any underlying resources. Called by Agent.close().
   * Optional — only relevant for stores with persistent handles (DB connections).
   */
  close?(): Promise<void>;
}

SqliteSessionStore (default)

The default store uses SQLite via the built-in bun:sqlite module. No native dependency to install, no prebuilt binary to fetch, no rebuild on platform mismatch. It is synchronous under the hood, fast, and zero-dep. The async interface on SessionStore still applies (the methods wrap synchronous calls in resolved promises).

Database file location

  • Default path: ${cwd}/.skawld/sessions.db.
  • Override via constructor option databasePath.
  • The store creates the directory if missing.
  • Schema is migrated on open (idempotent CREATE TABLE IF NOT EXISTS and small column-checks).

Schema

CREATE TABLE IF NOT EXISTS sessions (
  id          TEXT PRIMARY KEY,
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL,
  meta_json   TEXT NOT NULL DEFAULT '{}'
);

CREATE TABLE IF NOT EXISTS messages (
  session_id  TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  seq         INTEGER NOT NULL,
  appended_at TEXT NOT NULL,
  /** Full Message serialized as JSON. */
  message_json TEXT NOT NULL,
  PRIMARY KEY (session_id, seq)
);

CREATE INDEX IF NOT EXISTS idx_sessions_updated_at ON sessions(updated_at DESC);
CREATE INDEX IF NOT EXISTS idx_messages_session_seq ON messages(session_id, seq);

CREATE TABLE IF NOT EXISTS tasks (
  session_id     TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  id             TEXT NOT NULL,
  subject        TEXT NOT NULL,
  description    TEXT NOT NULL,
  active_form    TEXT,
  status         TEXT NOT NULL CHECK (status IN ('pending', 'in_progress', 'completed')),
  owner          TEXT,
  metadata_json  TEXT NOT NULL DEFAULT '{}',
  created_at     TEXT NOT NULL,
  updated_at     TEXT NOT NULL,
  PRIMARY KEY (session_id, id)
);

CREATE TABLE IF NOT EXISTS task_edges (
  session_id     TEXT NOT NULL,
  from_task_id   TEXT NOT NULL,
  to_task_id     TEXT NOT NULL,
  kind           TEXT NOT NULL CHECK (kind = 'blocks'),
  PRIMARY KEY (session_id, from_task_id, to_task_id),
  FOREIGN KEY (session_id, from_task_id) REFERENCES tasks(session_id, id) ON DELETE CASCADE,
  FOREIGN KEY (session_id, to_task_id) REFERENCES tasks(session_id, id) ON DELETE CASCADE
);

CREATE TABLE IF NOT EXISTS task_counters (
  session_id     TEXT PRIMARY KEY REFERENCES sessions(id) ON DELETE CASCADE,
  next_id        INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_tasks_session_status ON tasks(session_id, status);
CREATE INDEX IF NOT EXISTS idx_task_edges_to ON task_edges(session_id, to_task_id);

PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;

Implementation notes

  • Open the database in WAL mode for concurrent read performance and crash safety.
  • Use prepared statements for hot paths (appendMessages, loadMessages).
  • appendMessages runs in a transaction: compute next seq, insert messages, update sessions.updated_at, all atomic.
  • createTask runs in a transaction: read or initialize task_counters.next_id, insert the task, increment the counter, update sessions.updated_at. Deleted task ids are never reused.
  • updateTask applies scalar field changes, dependency edge changes, metadata merge/delete behavior, and updated_at inside one transaction. It rejects dependency cycles before commit.
  • delete(id) relies on foreign-key cascade to delete messages, tasks, task edges, and task counters.
  • Generated session ids use crypto.randomUUID(); generated task ids are numeric strings from task_counters.
  • Foreign keys must be enabled (off by default in SQLite).

InMemorySessionStore

A trivial store backed by a Map. Useful for tests and for SDK consumers who want ephemeral sessions (e.g. a web app that already has its own persistence layer and just wants the in-memory cache during a single HTTP request).

// src/sessions/memory.ts
export class InMemorySessionStore implements SessionStore {
  private sessions = new Map<string, SessionRecord>();
  private messages = new Map<string, StoredMessage[]>();
  private tasks = new Map<string, Task[]>();
  private taskCounters = new Map<string, number>();
  // ...implements the same interface, no persistence.
}

Resume semantics

  • Resuming loads all stored messages, in order, and rebuilds the in-memory conversation state. Compaction state is not stored; if the conversation grew large enough to be compacted earlier, the next run may re-compact when it crosses the threshold.
  • Task state is loaded from the same store as the session. Resuming a session resumes its tasks exactly as last written.
  • Tool-related state that is session-scoped but not in the message list or task tables (e.g. FileReadTracker) is reset on resume in v1. The model can re-Read files as needed.
  • The Agent that resumes a session does not have to be the same Agent instance that created it — what matters is that the same SessionStore is wired up.
  • The provider, model, and system prompt of the resuming Agent are used (i.e. they can differ from the originating Agent). This means a session can resume against a different model, which mostly works but may produce subtle drift when assistant messages contain provider-specific content blocks (most notably thinking). The loop tolerates this; provider translation drops blocks it cannot represent.
First-message ordering

The first message in any session must have role: "user". The store does not enforce this — the Agent does — but stores can rely on the invariant when filtering or restoring.

Usage example

import { Agent, AnthropicProvider, defaultTools } from "skawld";
import { SqliteSessionStore } from "skawld/sessions";

const agent = new Agent({
  provider: new AnthropicProvider(),
  model: "claude-sonnet-4-6",
  tools: defaultTools(),
  permissions: { mode: "default" },
  sessionStore: new SqliteSessionStore(),
  cwd: process.cwd(),
});

// New session
const a = await agent.session();
for await (const event of a.run("write a 'hello world' script in python")) { /* ... */ }

// Later, resume by id
const b = await agent.session({ id: a.id });
for await (const event of b.run("now turn it into a CLI with argparse")) { /* ... */ }

Implementing a custom store

To swap to Postgres, Mongo, or a managed service:

  1. Implement the SessionStore interface, including task persistence methods.
  2. Preserve message seq monotonicity per session, task id monotonicity per session, and the appendMessages/createTask/updateTask atomicity contracts.
  3. Honor updated_at updates on every append (used by list() ordering).
  4. Pass an instance to the Agent via sessionStore.

No engine change is required, and no other module needs to know about your store.

Build checklist

  • src/sessions/store.ts with SessionStore, SessionRecord, StoredMessage.
  • src/sessions/tasks.ts with Task, TaskStatus, CreateTaskInput, and TaskPatch.
  • src/sessions/sqlite.ts with SqliteSessionStore backed by bun:sqlite; WAL mode; idempotent migrations.
  • src/sessions/memory.ts with InMemorySessionStore.
  • Atomic appendMessages with transactional seq assignment.
  • Task tables: tasks, task_edges, and task_counters in the same SQLite database as sessions.
  • Task persistence methods on SessionStore: create, get, list, update, delete.
  • Task edge consistency and dependency cycle rejection.
  • Default DB path of ${cwd}/.skawld/sessions.db with directory creation.
  • list() ordered by updated_at DESC; limit / offset honored.
  • Foreign-key cascade on delete for messages, tasks, task edges, and counters.
  • Tests: create/load roundtrip; append assigns increasing seqs; cross-session isolation; list ordering; task create/list/get/update/delete; task ids are not reused; task edges survive resume; delete cascades; concurrent appends and task creates in the SQLite store (run in a tight loop within a single process).
  • Document the interface contract clearly so third-party stores can satisfy it.
  • src/sessions/index.ts exporting the interface and both default stores.
