Subagents

Spawn a focused child agent from within a tool call. A subagent runs as a fresh Session under the same Agent — same provider, same model, same MCP, same permission engine — but with its own message history, file tracker, and a filtered view of the parent's tools. The model invokes them via the new Subagent tool; the child's events stream back into the parent's event iterator wrapped as SubagentEvent.

Concept

Subagents and skills both let you compose model behavior into reusable units, but they differ on a single axis:

  • Skills (module 11) are inline-substituted prompt packs. The body of SKILL.md is returned to the model as a tool result; the model acts on it on its next turn within the same session.
  • Subagents (this module) spawn a real child Session with its own loop, its own history, and its own tool registry. The parent waits while the child runs and only sees the child's final assistant text — plus, optionally, every child event wrapped as SubagentEvent for live UI.

Subagents are intentionally narrow in v1: sync execution only. No background lifecycle, no notifications, no worktrees, no per-agent MCP servers, no per-agent permission modes, no per-agent model overrides, no fork-style context inheritance. The child always starts fresh; the parent always waits.

File format

An agent definition is a single markdown file: <configDir>/agents/<name>.md. (Flat layout — no per-agent subdirectory, unlike skills.) The file starts with a YAML frontmatter block; everything after the closing --- is the system prompt body.

---
name: code-reviewer
description: Review code for correctness, style, and security.
tools:
  - Read
  - Grep
  - Bash
---

You are a senior code reviewer. Read the changed files, walk each
acceptance criterion, and report findings ranked by severity. Be
concise: file:line refs only, no prose summaries.

Frontmatter fields

Key (YAML)TS fieldRequiredMeaning
descriptiondescriptionyesOne-line summary shown to the model in the Subagent tool's dynamic catalog.
namenamenoDefaults to the filename (minus .md). Must match /^[a-z0-9][a-z0-9_-]*$/i.
toolstoolsnoTool allowlist. Array or comma-separated string. Wildcard ["*"] or omitted = pass-through the parent's full tool registry. Names that don't resolve in the parent are silently dropped at spawn time. (...) permission patterns are stripped: "Bash(npm:*)" → "Bash".

Lenient frontmatter — Claude-compatible loading

The loader consumes only the three documented keys above. Every other key is silently ignored. This means a user can drop a Claude Code .claude/agents/*.md file straight into <configDir>/agents/ and it just works — the fields Skawld v1 doesn't support (model, provider, permissionMode, mcpServers, hooks, color, effort, maxTurns, memory, isolation, background, …) are dropped without warning.

---
name: from-claude
description: Ported from .claude/agents/ unchanged.
model: claude-opus-4-7      # ignored
provider: anthropic          # ignored
permissionMode: acceptEdits  # ignored
color: blue                  # ignored
mcpServers:                  # ignored
  - slack
hooks:                       # ignored
  SessionStart:
    - command: echo hi
tools:
  - Read
  - Grep
---

Body becomes the system prompt — exactly as in Claude.

Per-file failures (missing description, malformed YAML, name collision with another agent, IO errors, symlinks) become SkippedAgent entries — the loader never throws. The built-in default agent (see below) is always available even when zero disk agents are loadable.

Tools field normalization

The tools field accepts both array and comma-string forms, normalizing to a deterministic string[] at load time:

tools: [Read, Grep, Bash]          # → ["Read", "Grep", "Bash"]
tools: "Read, Grep, Bash"          # → ["Read", "Grep", "Bash"]
tools: ["Bash(npm:*)", "Read"]     # → ["Bash", "Read"]  (perm pattern stripped)
tools: 42                          # → undefined         (wrong type, lenient drop)
tools: ["Read", "DoesNotExist"]    # → loads, but at SPAWN time the unresolved name is dropped

An omitted tools field (or ["*"]) means “wildcard” — the child sees the parent's full tool registry directly, including Subagent (so recursion works). A non-wildcard filter that omits Subagent forbids the child from spawning further subagents.

The built-in default agent

Skawld ships one built-in subagent definition with the internal name _default. It has a wildcard tool list (full access) and a generic system prompt. It is not listed in the catalog the model sees — it's the implicit fallback used when the Subagent tool is called with no subagent_type.

When spawned, the default agent is given a synthetic display name Agent #N, where N is a per-parent-session counter that increments on each default spawn within that parent. Two independent parents each start at #1.

The Subagent tool

The Subagent tool is registered automatically on first agent.session() call (the same lazy-init step that connects MCP servers and loads skills). It has scope: "exec" and parallelSafe: true — multiple subagents can be in flight at once.

interface SubagentInput {
  description: string;       // Short 3-5 word UI label
  prompt: string;            // The task for the subagent (provide complete context)
  subagent_type?: string;    // Pick from the disk catalog; omit for the built-in default
}

The tool's description is dynamic — it lists every disk-loaded agent's name: description and tells the model to omit subagent_type for the general-purpose default:

Launch a subagent to handle a focused task.

Available subagent types:
- code-reviewer: Review code for correctness, style, and security.
- researcher: Research a question and report findings.

Omit `subagent_type` to spawn a general-purpose subagent with full tool access.
The subagent returns a single text response; its tool calls and partial output
stream into your event log while it runs.

On call, the tool resolves the definition, builds a filtered tool registry view, computes the display name, generates a fresh subagentRunId, and delegates to runSubagent(…). The result the parent model sees is one tool_result block whose content is the child's last assistant message's text. Unknown subagent_type, abort, or error all surface as is_error: true with a helpful message.

SubagentEvent

Every event the child Session emits is wrapped and pushed into the parent's event iterator via the new ToolContext.emit sink — wired into the scheduler's sequential writes path (which is where exec-scope tools run).

interface SubagentEvent {
  type: "subagent_event";
  parent_session_id: string;   // Parent — the Session the consumer iterates
  subagent_run_id: string;     // Unique per Subagent tool call
  subagent_type: string;       // 'code-reviewer' | '_default' | ...
  display_name: string;        // 'code-reviewer' | 'Agent #1' | 'Agent #2' | ...
  event: Event;                // The child's original event, untouched
}

The wrapped events appear in the parent's iterator between the tool_call_start and tool_call_end brackets for the Subagent tool call. A UI can render a hierarchical view by filtering on type === "subagent_event" and grouping by subagent_run_id.

Nested spawns produce nested wrappers: a grandchild's events arrive in the parent's iterator wrapped twice — the outer wrapper carries the child's ids, and the inner event field is itself a SubagentEvent carrying the grandchild's ids.

Recursion

Subagents can spawn subagents. The Subagent tool is just another tool in the registry; if it's in the child's filtered tool list (via wildcard or by explicit listing), the child can call it. There is no depth cap in v1.

To forbid recursion for a particular subagent, give it a non-wildcard tools filter that omits Subagent:

tools:
  - Read
  - Grep
  # No "Subagent" — child cannot spawn further subagents.

Lifecycle & persistence

  • Child Session. Constructed directly via new Session(...) (bypassing agent.session()) so MCP and skills are not re-connected — they're memoized on the parent Agent and already ran. The child is registered in agent.sessions for the duration of the run so a nested Subagent call (recursion) or a Skill call from within the child can resolve their session lookup; the runner unregisters in its finally block.
  • Tool registry. The child gets a SessionInternal.toolsOverride — either the parent's registry by reference (wildcard) or a fresh filtered copy. The loop and the scheduler read tools via si.toolsOverride ?? ai.tools at every site.
  • System prompt. The child gets a SessionInternal.systemBlocksOverride — a fresh buildSystemBlocks(...) output where the userInstructions block is replaced with the agent definition's body. Identity / tool-protocol / environment blocks remain identical to the parent's.
  • SQLite row. The child Session's row carries parent linkage in meta: { parentSessionId, subagentType, subagentRunId, displayName }. Use this to reconstruct a parent → children tree from the store.
  • Skill listing. Suppressed for subagent runs — the listing in the parent's history is already part of the parent's prompt cache; surfacing it again in a fresh subagent loop would just be noise. Gate: si.toolsOverride === undefined in the loop's first-user-message injection.
  • Abort propagation. The parent's tool-call AbortSignal is passed into Session.run(prompt, { signal }), so a pre-aborted signal is observed by the loop's anySignal guard on the first iteration. An additional one-shot listener on the signal also calls childSession.abort() so mid-stream aborts propagate.

Deferred to v2

The brainstorm explicitly cut the following features to keep v1 narrow and Claude-portable. Future work — none required for v1 correctness:

  • Per-agent model / provider override. Cross-provider subagent ("run this researcher on GPT-5 while the orchestrator runs on Opus") needs an AgentOptions.providers: Record<string, BaseProvider> registry plus provider: + model: frontmatter keys. Designed but not built.
  • Async / background lifecycle. No run_in_background, no notifications, no taskId + output_file handle. Sync only.
  • Per-agent permission mode, maxTurns, disallowedTools, MCP servers, skills preload, hooks, memory, worktree isolation, initialPrompt, effort, color, requiredMcpServers, criticalSystemReminder.
  • Fork-mode context inheritance. The child always starts with an empty conversation; no “inherit my context” mode.
  • SendMessage continuation. Every Subagent call is a fresh spawn; there is no way to resume a prior child.
  • Programmatic registration. No AgentOptions.agents field. The disk loader is the only source.
  • User-home overlay / parent-dir walk. Only <configDir>/agents/ is scanned. No ~/.skawld/agents/ fallback, no walk up the directory tree.
  • Depth cap. No protection against pathological recursion. Author your agent definitions responsibly.

Public API additions

Importable from "skawld":

  • SubagentEvent — the wrapper type added to the public Event union.

The Subagent tool itself is not directly importable — it's registered automatically by Agent.connectSubagents() on first session() call. The runner (runSubagent), the loader (loadAgentsFromDir), and the registry (buildAgentRegistry) are internal: subagent authoring happens via disk files, not code.
