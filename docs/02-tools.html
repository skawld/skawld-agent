Tools

The ten built-in tools that ship with skawld v1, plus the Tool interface they implement. Tools are pure functions over a context object; they do not know about providers or the agent loop; task tools use the session store provided in their context.

Tool interface

// src/tools/base.ts
import type { SessionStore } from "../sessions/store";

export interface ToolSchema {
  name: string;
  description: string;
  input_schema: {
    type: "object";
    properties: Record<string, JSONSchema>;
    required?: string[];
  };
}

/** Granular capability classification used by the permission engine and the parallel scheduler. */
export type ToolScope = "read" | "write" | "exec";

/** Read-only context passed to every tool execution. */
export interface ToolContext {
  /** Working directory for relative paths. */
  cwd: string;
  /** AbortSignal — tools must respect this and stop ASAP when fired. */
  signal: AbortSignal;
  /** Shared file-read tracker — required by Edit to enforce Read-before-Edit. */
  fileReadTracker: FileReadTracker;
  /** Session id, used by task tools and any session-scoped tool state. */
  sessionId: string;
  /** Run id, for logging/correlation. */
  runId: string;
  /** Persistent session store. Task tools use this for session-scoped task state. */
  sessionStore: SessionStore;
}

/** What a tool returns. */
export interface ToolResult {
  /** The content fed back to the model as a tool_result block. */
  content: string | Array<{ type: "text"; text: string } | { type: "image"; source: { type: "base64"; media_type: string; data: string } | { type: "url"; url: string } }>;
  /** Short human-readable summary for permission prompts & UIs. */
  summary: string;
  /** Tool execution itself errored (the model will see is_error: true). */
  is_error?: boolean;
}

export interface Tool<Input = Record<string, unknown>> {
  readonly name: string;
  readonly description: string;
  readonly input_schema: ToolSchema["input_schema"];
  /** Permission scope. Used by the scheduler and the permission engine. */
  readonly scope: ToolScope;
  /** True if this tool is safe to run concurrently with other tools of the same scope. */
  readonly parallelSafe: boolean;
  /** Validate & coerce raw input into a typed Input. Throw a ToolExecutionError on bad shape. */
  validate(raw: Record<string, unknown>): Input;
  /** Run the tool. */
  execute(input: Input, ctx: ToolContext): Promise<ToolResult>;
  /**
   * Produce a one-line summary of what this call will do, used in PermissionRequestEvent.
   * Called BEFORE execute and BEFORE permission resolution.
   */
  summarize(input: Input): string;
}

Parallel-execution rules

  • Tools with scope: "read" are always parallel-safe and run concurrently when the model emits multiple at once.
  • Tools with scope: "write" or scope: "exec" run sequentially in the order the model emitted them, even if parallelSafe: true, because order may matter for correctness.
  • The scheduler groups the model's tool calls per turn into batches by scope: one parallel batch of reads, then writes/execs serially.

Implementation principles

The tool API stays small, but the built-ins should be production-grade. Borrow operational lessons from Claude Code where they improve speed, correctness, or model ergonomics without importing UI, sandboxing, background tasks, subagents, hooks, or LSP integrations.

  • Prefer rg for file listing and content search. It is the fast path for both Glob and Grep.
  • Keep pure-Node fallbacks for portability when rg is missing. The fallback is slower but must preserve the same output shape.
  • Use structured OS APIs where possible: spawn/execFile with argument arrays, never shell interpolation for internal rg calls.
  • Honor ctx.signal in every tool. Long-running filesystem walks, rg calls, and shell commands must stop promptly when aborted.
  • Cap output intentionally. Large outputs should be truncated with an explicit marker rather than allowed to exhaust memory or flood context.
  • Keep permission policy outside tools. Tools expose scope, summaries, and path metadata; the permission engine decides whether a call may run.

Shared helpers

Implement these once and reuse them across tools instead of duplicating behavior:

  • resolvePath(inputPath, cwd): resolves relative paths against ctx.cwd, expands ~, and returns an absolute normalized path.
  • findExecutable(name): searches PATH once and caches the result. Used for rg.
  • runRipgrep(args, cwd, signal): wraps rg execution, treats exit code 1 as "no matches", supports timeout/abort, strips CRLF, and retries transient EAGAIN once with -j 1.
  • formatNumberedLines(content, startLine): emits stable cat -n-style line numbers.
  • truncateOutput(text, maxChars): preserves the beginning of output and appends a marker with the number of omitted characters.
  • atomicWriteFile(path, content): writes to a temp file in the same directory, preserves existing mode when possible, fsyncs the temp file, renames it into place, and cleans up temp files on failure.

Tool registry

// src/tools/registry.ts

export class ToolRegistry {
  private map = new Map<string, Tool>();

  register(tool: Tool): void {
    if (this.map.has(tool.name)) throw new ConfigError(`tool '${tool.name}' already registered`);
    this.map.set(tool.name, tool);
  }
  get(name: string): Tool | undefined { return this.map.get(name); }
  list(): Tool[] { return Array.from(this.map.values()); }
  schemas(): ToolSchema[] {
    return this.list().map(t => ({ name: t.name, description: t.description, input_schema: t.input_schema }));
  }
}

FileReadTracker (Read-before-Edit)

// src/tools/file-tracker.ts

/**
 * Tracks which files have been Read during the current session.
 * The Edit tool checks this and refuses to edit a file that has not been Read.
 * Cleared per-session, not per-run; Reads in earlier runs still count.
 */
export class FileReadTracker {
  private read = new Set<string>();
  /** Resolve to absolute, then add. */
  markRead(absPath: string): void { this.read.add(absPath); }
  hasRead(absPath: string): boolean { return this.read.has(absPath); }
}
Why this is enforced

The Edit tool requires an exact-string match between old_string and current file contents. Without Read-before-Edit, the model has nothing to anchor its old_string on except its own assumptions, which produces a stream of hallucinated edits. Enforcing Read first makes the model retrieve real bytes before attempting changes. This is a v1 hard rule.

Read

Reads a file from the local filesystem. The most-used tool in any agent loop.

// src/tools/read.ts

export interface ReadInput {
  file_path: string;        // required, absolute or relative to cwd
  offset?: number;          // 1-indexed line to start at (default 1)
  limit?: number;           // max lines to read (default 2000)
}

const SCHEMA = {
  type: "object",
  properties: {
    file_path: { type: "string", description: "Absolute or relative path to the file." },
    offset:    { type: "number", description: "1-indexed line to start at. Defaults to 1." },
    limit:     { type: "number", description: "Maximum number of lines to read. Defaults to 2000." },
  },
  required: ["file_path"],
} as const;

Behavior

  • Resolve file_path against ctx.cwd if not absolute.
  • Read with the configured offset / limit. Use a small-file fast path (readFile then split lines) and a streaming path for large files so range reads do not require loading the entire file into memory.
  • Format output with line numbers in cat -n style: " 42\tcontent".
  • Lines longer than 2000 characters are truncated with a marker.
  • Empty files return a single line: "<file is empty>".
  • On success, call ctx.fileReadTracker.markRead(absPath).
  • For binary files (detected via known binary extensions and null bytes in the first 8 KiB), return an error message rather than garbled text.
  • For PNG/JPEG/GIF/WebP: return an image content block instead of text.
  • Block device paths that would hang or produce unbounded output, such as /dev/zero, /dev/random, /dev/stdin, and /proc/*/fd/0.

Errors (returned as is_error: true, not thrown)

  • File does not exist.
  • Permission denied.
  • Path is a directory.

Scope: "read". parallelSafe: true.

Write

Creates a new file or overwrites an existing one.

export interface WriteInput {
  file_path: string;
  content: string;
}

const SCHEMA = {
  type: "object",
  properties: {
    file_path: { type: "string", description: "Absolute or relative path to write." },
    content:   { type: "string", description: "Full file contents." },
  },
  required: ["file_path", "content"],
} as const;

Behavior

  • If the file already exists and has not been Read in this session, refuse with is_error: true and a message instructing the model to Read first. (Same Read-before-modify principle as Edit, applied to overwrites.)
  • Create parent directories as needed.
  • Write the file with atomicWriteFile: temp file in the same directory, preserve existing mode when possible, fsync temp file, then rename.
  • After writing, call ctx.fileReadTracker.markRead(absPath) so subsequent Edits don't need a re-Read.
  • Return a short success summary including the byte count.

Scope: "write". parallelSafe: false.

Edit

Performs an exact string replacement in an existing file.

export interface EditInput {
  file_path: string;
  old_string: string;
  new_string: string;
  replace_all?: boolean;    // default false
}

const SCHEMA = {
  type: "object",
  properties: {
    file_path:   { type: "string" },
    old_string:  { type: "string", description: "Exact string to find. Must be unique unless replace_all is true." },
    new_string:  { type: "string", description: "String to replace it with." },
    replace_all: { type: "boolean", description: "Replace every occurrence. Defaults to false." },
  },
  required: ["file_path", "old_string", "new_string"],
} as const;

Behavior

  1. Resolve absolute path.
  2. If not in fileReadTracker → return is_error: true with message: "You must Read this file before editing it."
  3. If old_string === new_string → return is_error: true with message: "No changes to make: old_string and new_string are exactly the same."
  4. Read current contents.
  5. Reject very large files before loading them fully. v1 limit: 100 MiB for Edit; use Read + Write or Bash for larger generated artifacts.
  6. Count occurrences of old_string:
    • 0 occurrences → is_error: true, message: "old_string not found in file."
    • 1 occurrence → replace it.
    • >1 occurrences and replace_all: false → is_error: true, message: "old_string matches N occurrences; provide more surrounding context for a unique match, or pass replace_all: true."
    • >1 occurrences and replace_all: true → replace all.
  7. Preserve the file's existing line-ending style where feasible.
  8. Write the file with atomicWriteFile.
  9. Return a diff-style summary: "-3 +5 lines in foo.ts".
old_string must equal current bytes

The match is exact byte-for-byte, including whitespace and indentation. If the model is editing content it received from Read, the line-number prefix from cat -n formatting must not be part of old_string. Document this in the tool description so the model understands.

Scope: "write". parallelSafe: false.

Bash

Runs a shell command. No sandbox in v1 — runs in the user's environment.

export interface BashInput {
  command: string;
  timeout_ms?: number;      // default 120000 (2 min), max 1800000 (30 min)
  description?: string;     // optional 1-line description from the model for UIs
}

const SCHEMA = {
  type: "object",
  properties: {
    command:     { type: "string", description: "Shell command to execute." },
    timeout_ms:  { type: "number", description: "Timeout in milliseconds. Defaults to 120000, max 1800000." },
    description: { type: "string", description: "Brief description of what the command does." },
  },
  required: ["command"],
} as const;

Behavior

  • Spawn with child_process.spawn using the user's default shell (process.env.SHELL on POSIX, cmd.exe on Windows). Pass the command via -c (POSIX) or /c (Windows).
  • cwd = ctx.cwd (the agent's working directory).
  • env = inherited from the process. No scrubbing in v1.
  • Stream stdout and stderr into bounded accumulators instead of buffering unbounded child-process output.
  • Capture stdout and stderr separately, then format them clearly in the result.
  • Enforce timeout_ms (clamp to [100, 1800000]). Default is 120000 ms (2 minutes); max is 1800000 ms (30 minutes), matching long Claude Code-style build/test workflows.
  • On timeout, send SIGTERM to the process tree, wait 2s, then SIGKILL. Return is_error: true with the timeout reason.
  • If ctx.signal aborts, kill the process tree the same way.
  • Truncate combined output at 30000 characters with a clear "… (output truncated)" marker.
  • Return the combined output plus the exit code on the last line: "exit: 0".

Scope: "exec". parallelSafe: false.

Glob

Finds files by glob pattern.

export interface GlobInput {
  pattern: string;          // e.g. "src/**/*.ts"
  path?: string;            // root to search; default ctx.cwd
}

const SCHEMA = {
  type: "object",
  properties: {
    pattern: { type: "string", description: "Glob pattern, e.g. 'src/**/*.ts'." },
    path:    { type: "string", description: "Directory to search in. Defaults to the working directory." },
  },
  required: ["pattern"],
} as const;

Behavior

  • Primary path: use rg --files --glob <pattern> for speed, with cwd: path ?? ctx.cwd.
  • Fallback path: if rg is unavailable, use fast-glob with dot: false, onlyFiles: true, followSymbolicLinks: false.
  • Respect .gitignore by default in both paths. Do not traverse .git, .hg, .svn, or similar VCS metadata directories.
  • For absolute glob patterns, extract the static base directory before the first glob metacharacter and pass the remaining pattern to rg --glob / fast-glob.
  • Sort results by modification time (most recent first). Use a single stat pass and sort in memory.
  • Cap at 1000 results. If truncated, append a marker line.
  • Return paths as a newline-separated string, relative to path.
  • If zero matches, return "No matches found." as success (not an error).

Scope: "read". parallelSafe: true.

Grep

Searches file contents by regex. Prefers ripgrep (rg) for speed; falls back to a pure-Node implementation if rg is not on PATH.

export interface GrepInput {
  pattern: string;
  path?: string;                                // root to search; default ctx.cwd
  glob?: string;                                // file filter pattern
  type?: string;                                // file type alias (rg --type)
  output_mode?: "files_with_matches" | "content" | "count";  // default "files_with_matches"
  "-i"?: boolean;                               // case-insensitive
  "-n"?: boolean;                               // show line numbers (content mode)
  "-A"?: number; "-B"?: number; "-C"?: number;  // context lines (content mode)
  multiline?: boolean;                          // . matches newlines (content mode)
  head_limit?: number;                          // truncate output to N lines/entries; default 250
}

const SCHEMA = { /* mirror the above, all optional except pattern */ } as const;

Behavior — ripgrep path

  1. Detect rg at module load time or first use via findExecutable("rg"). Cache the result.
  2. If present, spawn rg with an argument array translated from the input. Do not invoke rg through a shell.
  3. Always pass a target path. Non-interactive rg can otherwise wait for stdin.
  4. Use --max-columns 500 to keep minified/base64 lines from dominating output. Exclude VCS metadata directories.
  5. Treat exit code 1 as success with no matches. Surface usage errors and permission errors as is_error: true.
  6. If rg fails with transient EAGAIN, retry once with -j 1.
  7. Capture stdout, apply head_limit by line in JS (rg has --max-count but at the per-file level).
  8. Return stdout as-is.

Behavior — JS fallback

  1. Walk the directory tree (respecting .gitignore if present, using ignore npm dep).
  2. Filter by glob / type if supplied.
  3. Compile the pattern as a RegExp with appropriate flags.
  4. Stream each file, matching line-by-line (or with bounded multiline buffering when multiline: true).
  5. Skip binary files using the same binary sniffing helper as Read.
  6. Produce the same output format the rg path produces, including -n line numbers, -A/-B/-C context lines, and output_mode handling.
  7. Apply head_limit.

Scope: "read". parallelSafe: true.

Treat ripgrep as the reference

Output format conformance with rg is important: the model is trained on grep/rg output shapes. The JS fallback should produce byte-identical output for the same input wherever feasible. Test the fallback against rg on a fixture corpus and assert equivalence for the common flag combinations.

Task State

Persistent, session-scoped planning state modeled after Claude Code's Tasks V2 shape, but intentionally smaller. These tools are the v1 planning interface. Task state lives in the SessionStore, not in the provider message history, so resumed sessions and future subagents share the same authoritative task list.

// src/sessions/tasks.ts
export type TaskStatus = "pending" | "in_progress" | "completed";

export interface Task {
  id: string;                 // "1", "2", "3" within one session
  session_id: string;
  subject: string;            // short title
  description: string;        // what needs to be done
  active_form?: string;       // present continuous form, e.g. "Running tests"
  status: TaskStatus;
  owner?: string;             // future subagent id/name
  blocks: string[];           // task ids this task blocks
  blocked_by: string[];       // task ids blocking this task
  metadata?: Record<string, unknown>;
  created_at: string;
  updated_at: string;
}

export interface CreateTaskInput {
  subject: string;
  description: string;
  active_form?: string;
  metadata?: Record<string, unknown>;
}

export interface TaskPatch {
  subject?: string;
  description?: string;
  active_form?: string;
  status?: TaskStatus | "deleted";
  owner?: string;
  add_blocks?: string[];
  add_blocked_by?: string[];
  remove_blocks?: string[];
  remove_blocked_by?: string[];
  metadata?: Record<string, unknown>;
}

Persistence contract

  • Tasks are stored in the same SessionStore as messages, using separate task tables in the SQLite default.
  • Task ids are monotonically increasing numeric strings scoped to one session. Deleted ids are not reused.
  • Deleting a session deletes all of that session's tasks.
  • Compaction does not delete or rewrite task state. The provider may see task tool results in history, but the database remains authoritative.
  • Task tools are auto-allowed by the permission engine. They modify skawld's session state, not project files.

Tool descriptions (the strings the model sees)

The description field is what the model reads when choosing whether and how to invoke a tool. Keep them precise, mention exact behaviors, and surface gotchas. Examples follow; phrase the final text however reads best, but cover these points.

// Read
"Reads a file from the local filesystem. Supports text and images (PNG/JPEG/GIF/WebP).
Use absolute paths or paths relative to the working directory. By default reads up
to 2000 lines from the start; use 'offset' (1-indexed) and 'limit' for larger files.
Lines longer than 2000 chars are truncated. Always Read a file before Editing it."

// Edit
"Performs an exact string replacement in a file. You must Read the file at least
once in this session before editing. The 'old_string' must appear exactly once
in the file unless 'replace_all' is true. Preserve indentation when matching."

// Bash
"Runs a shell command in the working directory. Default timeout is 2 minutes;
set timeout_ms for longer builds/tests up to 30 minutes. Quote paths with spaces.
There is no sandbox: the command runs with full user permissions."

// TaskCreate
"Creates a persistent task in the current session. Use for multi-step work,
planned follow-ups, or work that future subagents may coordinate on. Provide a
short subject and a concrete description."

// TaskList
"Lists persistent tasks for the current session. Use to check progress, find
available work, or see blockers before continuing a multi-step task."

// TaskGet
"Gets full details for one task. Use before materially changing description,
dependencies, owner, or metadata if your view may be stale."

// TaskUpdate
"Updates a persistent task. Can change status, subject, description, active_form,
owner, metadata, and dependency edges. Use status 'deleted' to remove a task."

Tool errors

Tool errors are returned in the ToolResult with is_error: true and fed back to the model as a tool_result block. The engine does not throw on tool errors. The only time a tool may throw a real ToolExecutionError is when something prevents the engine from recovering — e.g. a permission system bug or a corrupted registry. In normal operation, the model sees the error and chooses the next action.

Result formatting

String content is preferred where reasonable. Use the structured array form only when:

  • An image needs to come back (Read on PNG, future screenshot tool).
  • You need to interleave text and an image.
